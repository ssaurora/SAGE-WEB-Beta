## 一、总目标

下一轮不是继续“把更多页面改成 Decision Zone / Evidence Zone”，而是完成三件事：

1. 让 **Workbench** 真正按状态变形，而不是只做显隐调整。
2. 让 **Governance** 的证据区进入“分层阅读”模式，而不是继续纵向排块。
3. 让 **Overview / Results / Assets** 摆脱“术语套模板”，回到各自真实职责。

- Governance 的证据区已开始从纵向排块改为 tab 阅读结构。
- Workbench 已完成三套状态版式的首轮落地，Input / Runtime / Result 三种主布局开始真正分化。
- Workbench 的主动作已进一步收紧到 State Focus，地图区相关跳转已降级为次级关联链接。

- Governance 的 Blocking Evidence 阅读顺序已固定为 Required Actions → Missing Inputs → Failure Summary → Suggested Fixes。

- 当前代码侧已完成 `from` 字符串收口（主链路 + 关键支撑页），并通过构建验证。

## 二、P0：必须优先完成

# P0-1 Workbench：三套状态版式落地

**页面**：`components/pages/workbench-map-interactive.tsx`
**当前问题**：已经完成输入折叠、图层后置、对象检查折叠、主动作切换，但页面骨架仍基本固定，状态只影响内容，不真正改写布局。

### 任务

把 Workbench 拆成三种主布局模式：

#### A. Input Recovery Layout

适用状态：

- Waiting for Required Input
- Action Required
- Failed Recoverable

要求：

- `Inputs` 成为最大操作区
- `State Focus` 与输入修复动作并成一体
- `Map Canvas` 降级为辅助
- `View Results` 完全退出主视觉

#### B. Runtime Monitoring Layout

适用状态：

- Understanding
- Queued
- Running
- Processing Results

要求：

- `State Focus` 占主中心
- `Map Canvas` 作为运行观察画布
- `Inputs` 仅保留摘要
- `Execution Context` 继续折叠
- `Object Inspector` 不得默认展开

#### C. Result Transition Layout

适用状态：

- Completed

要求：

- `Result Ready` 升级为页面中心
- `View Results` 成为唯一主动作
- 地图重新成为结果主画布
- Inputs / Execution Context 均退后

### 开发要求

- 不要继续只靠 `if (...) render block` 微调高度
- 直接抽出三个布局函数或三个状态 layout component
- 状态决定主容器结构，而不是只决定局部内容

### 验收标准

- 用户一眼能看出当前是“修输入 / 看运行 / 看结果”哪一种页面
- 三种状态下页面结构明显不同，不是同一页面换文案
