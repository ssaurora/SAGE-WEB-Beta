## 1. 文档目的

本文用于指导当前 SAGE 前端从“功能堆叠型页面集合”重构为“任务驱动型科研工作台”。

本次重构不是一次普通的视觉美化，也不是单纯的样式统一，而是一次围绕产品认知逻辑进行的界面重建。

当前仓库已经明确呈现出产品定位：

- 根布局描述为 **Scene-first analysis workbench**
- 全局壳层强调 **Scene-first · GIS-first · Governance-visible**
- 导航结构同时存在全局对象与 Scene 内对象两层体系

这意味着前端不应继续按照“页面逐个装配”的方式生长，而必须围绕以下主线重构：

1. 用户通过 Scene 进入分析任务；
2. 系统在 Workbench 中组织输入、状态、运行与结果；
3. 治理信息必须可见，而不能沉没为后台字段；
4. 结果页必须服务于科研判断，而不是字段陈列。

---

## 2. 本次重构要解决的根问题

当前问题不是“页面不好看”，而是“前端没有把系统能力组织成可理解的工作环境”。

### 2.1 现有问题的本质

当前前端的问题主要体现在以下四个层面：

#### （1）产品主线不清

虽然系统内部已经形成 Scene、Task、Governance、Results、Assets 等对象体系，但这些对象在前端中仍被以并列页面和卡片块的方式暴露，导致用户无法快速判断：

- 自己当前是在操作场景，还是在处理任务；
- 当前页面是“概览页”“工作台页”还是“治理决策页”；
- Results、Reports、Task Governance 与 Scene 的关系；
- 哪些信息是主路径信息，哪些只是支撑证据。

#### （2）结构化事实没有成为界面中心

当前多个页面都存在“卡片并列 + 信息平铺”的倾向：

- Scenes 列表页仍是搜索、筛选、统计块和场景卡的并列堆放
- Scene Overview 仍是摘要、待办、活动、地图预览的传统拼装式概览
- Task Governance 页面以纵向卡片持续累积治理信息、manifest、event、audit，缺乏明确的决策区和证据区分离

这使页面虽然“信息很多”，但不构成“任务认知”。

#### （3）Workbench 被误解为地图页

当前 Workbench 实际承担的是复合任务控制台职责，包括：

- 权限态区分；
- 输入绑定与校验；
- 图层与地图交互；
- 运行状态与进度；
- 建议动作与上下文；
- 治理跳转与结果跳转

因此 Workbench 的本质不是“地图为主的信息展示页”，而是“分析执行控制台 + 空间画布”的混合工作台。若继续把地图设为绝对中心，会错误压缩任务控制语义。

#### （4）页面按模块组织，而不是按状态组织

当前系统内部已经存在明确状态语义，如：

- Understanding
- Waiting for Required Input
- Action Required
- Queued
- Running
- Processing Results
- Completed
- Failed

这些状态已经直接影响按钮、只读性、提示语和跳转逻辑  
但现有前端仍主要按“页面模块摆放”组织界面，而不是按“当前状态驱动界面重点”组织。

这是当前前端可理解性不足的根源。

## 3. 重构总目标

本次重构的目标不是“更现代”，而是“更可理解、更可决策、更符合科研任务流”。

### 3.1 核心目标

将前端重构为：

- **Scene-first 的任务工作台**
- **状态驱动的单屏操作环境**
- **治理可见的分析执行界面**
- **结论优先的科研结果消费界面**

### 3.2 用户体验目标

重构完成后，用户应能够：

1. 在进入任一核心页面后，3 秒内判断该页面当前属于哪一类任务阶段；
2. 在首屏内识别当前状态、阻塞原因与下一步动作；
3. 在不阅读全部字段的情况下完成关键判断；
4. 在 Workbench、Governance、Results 三个页面之间形成稳定认知，而不是迷失在对象跳转中。

### 3.3 产品目标

重构后的前端必须明确表达：

- Scene 是任务容器；
- Workbench 是执行控制台，而不是普通地图页；
- Governance 是决策与恢复页，而不是资料页；
- Results 是科研判断页，而不是结果列表页。

---

## 4. 重构原则

## 4.1 任务主线优先原则

所有页面都必须服务于同一条主线：

> 发起任务 → 补齐输入 → 运行分析 → 治理追踪 → 结果判断

任何不能支撑这条主线的信息，都必须降级或删除。

---

## 4.2 状态驱动原则

页面的主焦点不由组件类型决定，而由当前任务状态决定。

例如：

- 在 Waiting for Required Input 阶段，输入缺失与绑定修复必须成为首屏中心；
- 在 Running 阶段，运行进度与不可编辑事实必须成为首屏中心；
- 在 Failed Recoverable 阶段，阻塞原因与恢复动作必须成为首屏中心；
- 在 Completed 阶段，结果摘要与结果跳转必须成为首屏中心。

因此，“地图是否最大”“卡片是否居中”不是优先问题，“当前状态是否接管界面”才是优先问题。

---

## 4.3 决策优先原则

核心页面必须优先回答以下问题：

1. 当前处于什么状态；
2. 为什么处于这个状态；
3. 用户现在该做什么；
4. 系统依据什么给出这个判断。

顺序必须固定为：

1. 状态
2. 动作
3. 原因
4. 证据
5. 详情

不允许再以字段列表、卡片网格、事件流水自然堆叠替代这套顺序。

---

## 4.4 证据后置原则

Manifest、Lifecycle Events、Artifacts、Audit、Raw Fields 等内容是必要信息，但不是首屏主信息。

这些内容必须作为“证据区”存在，而不能与“行动区”处于相同视觉等级。

---

## 4.5 删除优先原则

本次重构不是“把现有内容收进更多面板”，而是先删、再并、再重排。

必须优先执行：

- 删除重复提示；
- 删除低价值统计块；
- 删除装饰性 badge；
- 删除无主线作用的辅助说明；
- 删除壳层中过重的视觉占位。

---

## 4.6 科研工作台原则

本系统是科研分析系统，不是市场化 AI 展示页，也不是驾驶舱大屏。

视觉上应体现：

- 克制；
- 稳定；
- 可阅读；
- 状态色有明确语义；
- 重点依靠层级、间距和密度控制，而不是依靠装饰。

---

## 5. 信息架构重构

## 5.1 重新定义一级对象

前端一级对象应明确为以下四类：

### （1）Scene

作为分析任务容器，承载主题、空间范围、场景资产和相关任务入口。

### （2）Workbench

作为分析执行控制台，承载输入准备、空间画布、运行控制、当前状态和相关跳转。

### （3）Governance

作为治理决策页，承载任务状态、阻塞原因、恢复建议、治理事实和审计证据。

### （4）Results

作为科研结果消费页，承载结论、地图结果、指标、解释、下载和结果可信判断。

---

## 5.2 顶层导航调整原则

当前全局导航将 `Scenes / Tasks / Task Governance / Assets / Reports / Settings` 并列展示  
该做法存在明显问题：对象层级混乱，容易让用户误把二级对象当一级对象。

本次重构要求：

- 顶层导航仅保留全局稳定对象；
- Scene 内对象不再和全局对象平级竞争；
- Task Governance 不应作为普通一级导航主入口长期暴露；
- Reports 与 Scene Results 的关系必须重新定义，避免双重结果入口。

### 调整建议

建议顶层导航收敛为：

- Scenes
- Assets
- Results（若确有跨 Scene 结果消费需求）
- Settings

其中：

- Task Governance 改为任务上下文驱动入口；
- Task Runs 改为 Scene 内二级视图；
- Audit 作为证据层入口，不抢顶层导航权。

---

## 5.3 页面切换语义重构

必须明确：

- Scene 内切换，是任务上下文保持下的局部视图切换；
- Governance 跳转，是从执行页进入治理决策页；
- Results 跳转，是从运行完成态进入结果判断页；
- Audit、Manifest、Artifacts 属于证据阅读，不属于主链路页面。

这意味着页面切换必须服务于任务链路，而不是形成“功能菜单式漫游”。

---

## 6. 全局壳层重构要求

当前 `AppShell` 采用左侧大面积卡片式导航 + 顶部页头的后台壳结构  
这套结构本身就在强化“拼装后台感”。

## 6.1 壳层问题

当前壳层存在以下问题：

- 左侧导航块过重；
- “Current page”提示价值极低；
- 页头再次重复页面名称；
- 导航卡片与内容卡片视觉语法一致，导致层级混乱；
- 壳层本身就在抢页面注意力。

## 6.2 壳层改造要求

### 必做项

1. 左侧导航去卡片化  
   改为真正的导航条，不再用大块边框卡片模拟入口。
2. 删除“Current page”模块  
   该模块不提供有效任务价值，应直接移除。
3. 页头瘦身  
   页头只保留必要定位信息与少量全局动作，不应重复页面标题。
4. 降低壳层存在感  
   壳层负责承载，不负责表达页面主内容。
5. 全局样式收敛  
   壳层、页面、卡片、面板必须具备明确层级差异，不允许全部使用统一卡片语法。

## 7. 状态驱动界面规范

本系统的核心界面必须按状态组织，而不是按栏目组织。

## 7.1 状态分类

建议前端统一支持以下主要状态语义：

- Draft / Idle
- Understanding
- Waiting for Required Input
- Action Required
- Queued
- Running
- Processing Results
- Completed
- Failed Recoverable
- Failed Terminal
- Cancelled

## 7.2 状态与界面中心关系

### Draft / Idle

界面中心：任务发起、输入准备、推荐操作。

### Understanding

界面中心：系统正在理解请求，用户暂不应被过量信息打扰。

### Waiting for Required Input

界面中心：缺失输入、绑定入口、约束说明、可执行条件。

### Action Required

界面中心：非法绑定、修复建议、恢复动作。

### Queued

界面中心：队列状态、只读提示、预计进入运行。

### Running

界面中心：运行进度、当前阶段、只读锁定、治理入口。

### Processing Results

界面中心：结果处理中提示、已完成阶段、尚不可下载项。

### Completed

界面中心：结论摘要、结果入口、结果消费。

### Failed Recoverable

界面中心：失败原因、建议修复、恢复主按钮。

### Failed Terminal

界面中心：失败说明、不可恢复说明、重新发起入口。

### Cancelled

界面中心：取消事实、是否允许重建任务。

## 8. 页面级重构方案

# 8.1 Scenes 列表页重构

当前 Scenes 列表页混合了：

- 权限状态 badge
- Create Scene
- 搜索、筛选、排序
- 总数、活跃/归档、任务数、结果数、访问模式
- 场景入口卡片

这是一种典型的“能放的都放”。

## 8.1.1 重构目标

Scenes 页应被重构为“进入任务空间的入口页”，而不是“场景统计展示页”。

## 8.1.2 必删项

必须删除或降级以下内容：

- 低价值统计块的成片展示；
- 访问模式这种低频说明块；
- 权限 badge 的重复装饰化表达；
- 对场景入口无直接帮助的冗余概览数据。

## 8.1.3 首屏结构

建议首屏只保留：

- 页面标题与一句产品定位说明；
- Create Scene 主按钮；
- 一个紧凑搜索框；
- 一个主要筛选器；
- Scene 列表入口区。

## 8.1.4 Scene 卡片要求

Scene 卡片只应保留：

- 场景名；
- 简要描述；
- 分析主题；
- 最近更新时间；
- 当前活跃状态。

任务数、结果数可保留其一，不应双双高调出现。

---

# 8.2 Scene Overview 重构

当前 Overview 仍然是“摘要 + latest task + latest result + 小地图 + pending + activities”的标准概览拼法

## 8.2.1 重构目标

Overview 应从“信息概览页”改为“Scene 当前状态入口页”。

## 8.2.2 首屏必须回答的问题

- 这个 Scene 当前最重要的任务状态是什么；
- 当前用户最该进入哪条链路；
- 当前是否有待处理事项；
- 最近的结果是否值得继续查看。

## 8.2.3 首屏结构建议

### 顶部主摘要区

- Scene 名称
- 分析主题
- 最近更新时间
- 当前最重要状态摘要

### 中央主行动区

- 继续进入 Workbench
- 查看当前治理状态
- 查看最近结果

### 辅助信息区

- Pending Actions
- Recent Activities

### 地图预览降级

地图预览仅作为上下文提示，不得与主行动区竞争。

---

# 8.3 Workbench 重构

Workbench 是本次重构的核心。

当前 Workbench 实际已经承载分析执行控制台职责，而不是普通地图视图

## 8.3.1 重构定位

Workbench 定位为：

> 以任务状态驱动的分析执行控制台  
> 地图是工作台的一部分，不是工作台本身

## 8.3.2 严禁继续使用的错误原则

以下原则必须废止：

- “地图永远是绝对主视觉”
- “所有工作台状态都使用同一布局”
- “输入、图层、状态、分析建议全部并列呈现”

## 8.3.3 正确原则

Workbench 的主视觉与布局必须随状态变化：

### 输入未齐 / 待修复阶段

主中心应为：

- Required Inputs 准备度
- 缺失项
- 无效绑定
- 绑定操作
- 可执行条件

地图退居为辅助上下文。

### Running / Queued 阶段

主中心应为：

- 当前状态
- 运行进度
- 当前锁定说明
- 治理入口
- 可查看但不可修改的画布与图层

### Completed 阶段

主中心应为：

- 结果摘要
- 查看结果主动作
- 地图结果联动
- 结果相关对象跳转

## 8.3.4 工作台布局建议

建议采用三段式，但不是固定语义三栏：

### 左侧：上下文与输入控制

- 输入准备
- 绑定/解绑
- 资源选择
- 图层基础控制

### 中部：动态主区

根据状态切换为：

- 输入修复中心
- 运行进度中心
- 地图结果中心

### 右侧：状态与跳转

- 当前状态摘要
- Suggested Next Steps
- Governance 入口
- Result 入口
- Object summary（按需展开）

## 8.3.5 必删与必降级项

必须处理以下问题：

1. 不再让每一块内容都变成完整 Card
2. Object Inspector 不应永久高权重常驻
3. Analysis / Task / Context 不能继续作为大杂烩卡片存在
4. Runtime Progress 必须提升为真正状态层，而不是附属进度条
5. “Workbench Permission Scope”这类说明型模块必须弱化，不得抢任务中心

---

# 8.4 Task Governance 重构

当前 Task Governance 页面最大的问题，不是信息不全，而是没有把“决策”和“证据”切开

## 8.4.1 重构定位

Task Governance 不是资料页，而是：

> 任务状态判断与恢复决策页

## 8.4.2 首屏结构必须重组为两大区

### A. 决策区

必须放在首屏最前：

- 当前状态
- 是否可恢复 / 可取消 / 可继续
- 当前阻塞原因
- 推荐动作
- 主按钮

### B. 证据区

作为后续区域或 tab：

- Missing Inputs
- Invalid Bindings
- Manifest Snapshot
- Lifecycle Events
- Artifacts
- Audit Summary

## 8.4.3 必删与必改项

1. 不允许继续把所有治理模块纵向平铺
2. Navigation Context 必须弱化
3. Governance Access 这类说明块不得高于决策信息
4. Lifecycle Events 不得首屏全量展开
5. Manifest 必须作为只读证据视图，不得抢占行动区位置

## 8.4.4 行动语义要求

Governance 页必须确保用户在首屏能一眼知道：

- 现在是不是轮到自己处理；
- 是该补输入、修绑定、恢复任务，还是查看结果；
- 如果失败，失败是可恢复还是终止型失败。

---

# 8.5 Results 重构

当前 Results 页的方向已提示“地图化、指标化与解释化结果消费”  
但结果页仍需明确从“列表页”转向“判断页”。

## 8.5.1 重构定位

Results 页应定位为：

> 科研结果判断与结果消费页

## 8.5.2 结果页必须优先回答的问题

1. 这轮结果对应哪个任务；
2. 当前结果是正式结果、处理中结果还是残留结果；
3. 最核心结论是什么；
4. 哪些地图、哪些指标、哪些解释互相对应；
5. 哪些文件值得下载。

## 8.5.3 布局建议

### 顶部结论区

- 结果标题
- 来源任务
- 结果状态
- 一句话结论

### 中部主证据区

- 地图结果
- 核心指标
- 指标与地图联动说明

### 下部消费区

- 解释文本
- 下载
- 原始字段或附加数据

## 8.5.4 必须避免的问题

- 不要先让用户看列表结构，再自己找重点；
- 不要把原始字段和核心结论放在同一视觉等级；
- 不要把下载区提前于结论区。

---

# 8.6 Assets 重构

Assets 目前容易沦为后台清单页。

## 8.6.1 重构目标

Assets 必须表达“资产在分析任务中的角色”，而不是“文件仓库”。

## 8.6.2 页面策略

### Scene 内 Assets

重点表达：

- 本场景当前可用资源
- 资源绑定状态
- 哪些资源参与当前分析

### 全局 Assets

重点表达：

- 可复用性
- 归属关系
- 绑定关系
- 可用于哪些 Scene 或任务

---

## 9. 视觉与组件重构要求

## 9.1 视觉基调

视觉风格必须体现：

- 科研工作台气质；
- 信息克制；
- 结构稳定；
- 不追求花哨 AI 感；
- 不追求大屏驾驶舱感。

## 9.2 颜色原则

颜色服务于状态语义，不服务于装饰。

建议仅保留以下语义色体系：

- 主色：主动作 / 当前焦点
- 成功色：完成 / 可消费
- 警告色：待处理 / 风险
- 危险色：失败 / 阻断
- 中性色：背景 / 辅助 / 说明

禁止再出现：

- 仅为好看而存在的大面积强调色；
- 无语义的状态堆色；
- Badge 过度使用。

## 9.3 卡片使用规范

Card 只能用于明确边界的内容块，不得用于一切内容。

必须执行：

- 导航去卡片化；
- 说明类模块弱卡片化；
- 主内容区尽量改为面板结构；
- 同级信息不允许通过“多卡片拼盘”表达层级。

## 9.4 按钮层级规范

按钮必须分为：

- 主按钮：唯一下一步动作
- 次按钮：可选但常见动作
- 弱按钮：辅助查看或跳转

禁止在一个局部区域内同时出现多个主按钮。

## 9.5 文本层级规范

- H1 每页仅一个；
- 页面级说明只保留一句；
- 说明文字必须少于行动信息；
- Badge 不承担复杂语义传达。

---

## 10. 单屏与滚动策略

## 10.1 单屏的正确定义

单屏不是“所有内容都塞进首屏”，而是：

> 在首屏完成当前阶段的判断与行动决策

因此：

- 首屏负责决策；
- 下层负责证据；
- 详情通过局部展开、tab 或次级区域提供。

## 10.2 滚动规则

允许：

- 面板内部滚动；
- 次级证据区滚动；
- 切换 tab 查看证据。

不允许：

- 核心任务页靠长页面向下寻找主动作；
- 首屏决策信息沉没在多屏之后；
- 用滚动替代结构设计。

## 11. 实施阶段

## Phase A：产品结构重构

目标：先修认知结构，不先修皮肤。

工作项：

- 明确 Scene / Workbench / Governance / Results 四类页面定位；
- 重画全局导航与 Scene 内导航关系；
- 重新定义主链路页面跳转语义；
- 输出统一页面骨架图。

验收标准：

- 每个页面都能用一句话准确说明自身任务；
- 顶层与次级对象不再混淆；
- 主链路清晰可复述。

---

## Phase B：壳层与全局样式收敛

工作项：

- 改造 AppShell；
- 导航去卡片化；
- 删除 Current page；
- 页头瘦身；
- 建立统一间距、标题、按钮、状态样式体系。

验收标准：

- 壳层不再抢占页面主视觉；
- 全局视觉语言统一；
- 页面不再显著呈现后台拼装感。

---

## Phase C：Workbench / Governance 核心页重构

工作项：

- Workbench 改为状态驱动布局；
- Governance 改为决策区 + 证据区结构；
- 统一运行态、待输入态、待修复态、完成态的界面差异；
- 统一主动作表达。

验收标准：

- 首屏能够一眼看出当前状态和下一步动作；
- Workbench 不再是地图中心拼装页；
- Governance 不再是资料堆叠页。

---

## Phase D：Overview / Results / Assets 收口

工作项：

- Overview 改为 Scene 状态入口；
- Results 改为结论优先的判断页；
- Assets 改为角色导向的资源页；
- 补齐空态、错态、加载态。

验收标准：

- 各页面职责边界清晰；
- 结果页具备科研判断价值；
- 资产页具备任务关联价值。

---

## Phase E：统一走查与微调

工作项：

- 审查所有页面是否存在重复信息；
- 审查状态色与按钮语义是否一致；
- 审查单屏是否真正完成决策；
- 审查页面是否仍残留“后台拼装感”。

验收标准：

- 页面首屏都能完成判断；
- 视觉噪音显著下降；
- 主任务链路可顺畅走通。

## 13. 验收标准

## 13.1 产品维度

- 用户能明确区分 Scene、Workbench、Governance、Results 的职责；
- 页面不再像功能菜单集合；
- 用户能快速知道当前轮到谁行动。

## 13.2 交互维度

- 首屏内必须存在唯一明确主动作；
- 页面不依赖长滚动寻找关键操作；
- 状态变化会真实改变页面主焦点。

## 13.3 视觉维度

- 页面不再呈现拼装式卡片后台感；
- 壳层不再与内容抢视觉；
- 配色、层级、按钮、状态语义统一。

## 13.4 科研使用维度

- 结果页能支持科研判断；
- 治理页能支持恢复决策；
- 工作台能支持任务执行，而不是只是展示地图。

---

## 14. 结论

当前 SAGE 前端的核心问题，不是缺少页面，而是缺少围绕任务主线的界面组织。

因此，下一轮前端工作不得再以“补页面”“补模块”“补卡片”为主，而必须按照以下顺序推进：

1. 先重建产品信息架构；
2. 再重建状态驱动界面；
3. 再重构 Workbench 与 Governance；
4. 最后统一视觉语言。

本次重构的目标，不是把前端从“能看”改成“更漂亮”，而是把它从“页面堆叠”改成“科研任务工作台”。
