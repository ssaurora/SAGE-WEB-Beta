### 2.1 P0-A：Workbench 输入绑定动作闭环

**优先级**：🔴 最高  
**依赖**：无  
**工作量**：2-3 天

#### 2.1.1 需求定义

现状：
- ✅ Inputs 面板展示 Required/Optional 列表
- ❌ 缺真实的上传 / 绑定 / 替换 / 移除 动作

目标状态：
- [ ] Upload Asset 流程（选文件 → 上传 → 转换为已上传状态）
- [ ] Bind 动作（从已上传列表绑定到 Required Inputs）
- [ ] Replace 动作（替换已绑定的资源）
- [ ] Remove 动作（解除绑定）
- [ ] Invalid Binding 高亮与快速修复路径
- [ ] 绑定后 UI 反馈（绑定状态色彩、次序调整等）

#### 2.1.2 实现清单

- [ ] 创建 `components/pages/inputs-panel-interactive.tsx` 
  - 功能：可交互的 Inputs 面板，替代当前静态版本
  - 包含：Upload Modal、Bind Select、Replace 确认、Remove 确认
  
- [ ] 创建 `lib/actions/asset-binding.ts`
  - 上传资源 API 调用
  - 绑定 / 替换 / 移除 的服务端调用封装

- [ ] 修改 `components/pages/workbench-map-interactive.tsx`
  - 集成 InputsPanelInteractive
  - 监听绑定状态变化，更新 `requiredInputs` / `optionalInputs` 状态
  - 绑定完成后更新 `canRunByInputs` 状态

- [ ] 创建上传/绑定相关的 Modal 组件
  - AssetUploadModal
  - BindInputModal
  - ReplaceConfirmDialog
  - RemoveConfirmDialog

#### 2.1.3 验收标准

```gherkin
Scenario: 用户上传并绑定输入
  Given 用户在 Workbench -inputs 区
  When 点击"+"按钮
  Then 弹出 Upload Modal
  
Scenario: 用户选择已上传资源进行绑定
  Given 有已上传未绑定的资源
  When 在 Required Input 行点击"Bind"
  Then 弹出 Select 菜单列出可用资源
  
Scenario: 绑定完成后输入状态更新
  Given 用户完成了资源绑定
  Then Inputs 面板中该输入状态变为"Bound"
  And 计数器"Required Inputs Ready"增加
  And 若所有必需输入都 Bound，"Run Analysis"按钮启用
```

### 2.3 P0-C：Governance 结构补齐与失败恢复路径

**优先级**：🔴 高  
**依赖**：无  
**工作量**：1.5-2 天  
**影响范围**：Task Governance 页面

#### 2.3.1 需求定义

现状：
- ✅ Summary / Events / Artifacts / Audit 摘要区存在
- ❌ 缺 Manifest Tab（完整参数只读视图）
- ❌ 失败恢复路径表现不够清晰（如何 Fix and Resume？）
- ❌ 取消后的重新发起路径不清晰

目标状态：
- [ ] Manifest Tab（可展开的参数树，区分只读参数 / 可修改参数 / 说明注释）
- [ ] Governance Panel 强化
  - Failed Recoverable 状态：高亮问题输入 / 绑定，展示"What's Needed to Resume"
  - Failed Terminal 状态：明确表示"需重新发起分析"
  - Cancelled 状态：明确表示"可重新运行该分析"
- [ ] 快速修复与恢复按钮
  - "Fix and Resume"（跳转回 Workbench 并高亮问题输入）
  - "Start New Analysis"（从问题点重新开始）
  - "Re-run"（完全重新执行）

#### 2.3.2 实现清单

- [ ] 创建 `components/pages/governance-manifest-tab.tsx`
  - 展示完整 Manifest（参数、输入、绑定）
  - 参数树，可展开/折叠
  - 区分编辑参数 vs 只读参数

- [ ] 创建 `components/pages/governance-recovery-panel.tsx`
  - 根据失败类型展示不同的恢复建议
  - 包含 Failed Recoverable / Failed Terminal / Cancelled 三种分支
  - 按钮：Fix and Resume / Start New Analysis / Re-run

- [ ] 修改 `/app/task-governance/[taskId]/page.tsx`
  - 添加 Manifest Tab 到主容器
  - 集成 GovernanceRecoveryPanel

- [ ] 修改 TaskGovernanceActions（如果还存在）
  - 融合恢复路径，使按钮逻辑更清晰

#### 2.3.3 验收标准

```gherkin
Scenario: 查看失败任务的完整 Manifest
  Given 任务处于 Failed 状态
  When 点击 Governance 页的"Manifest"Tab
  Then 显示完整参数树，每个参数有类型 / 值 / 说明

Scenario: Failed Recoverable 恢复路径
  Given 任务处于 Failed Recoverable 状态
  When 查看 Governance 页
  Then 显示"What's Needed to Resume"建议区
  And "Fix and Resume"按钮可用
  
Scenario: Failed Terminal 不可恢复路径
  Given 任务处于 Failed Terminal 状态
  When 查看 Governance 页
  Then 显示"需重新发起"的明确说明
  And "Start New Analysis"按钮可用（回到 Workbench）
```

## 6. 风险控制

### 风险 1：绑定流程与后端模型不匹配
**应对**：前置完成内部协议定义（上传返回 Asset ID、绑定时需提供哪些字段等）

### 风险 2：Reports 模型与 Results 模型混淆
**应对**：明确 Report = Results + Interpretation + Manifest Summary，不是简单别名

### 风险 3：Governance 失败恢复按钮逻辑复杂
**应对**：绘制决策树，明确 3 种失败类型 → 对应 3 种按钮组合

### 风险 4：时间不够完成所有工作
**应对**：P0-A 和 P0-D 是最关键，其次是 P0-B，P0-C 可适度后延
