## 1. 文档目的

本文用于把既有前端设计规范，进一步转换为可执行的页面验收清单。

目标是回答：

1. 页面是否真的实现了设计文档要求。
    
2. 页面在关键状态下是否表现正确。
    
3. 工作台、治理页、结果页是否真正承接了结构化对象，而不是只做了静态 UI。
    
4. 设计、前端、产品在评审时应检查哪些点。
    

本文适用于：

- 设计验收
    
- 前端自测
    
- 产品评审
    
- 里程碑验收
    

---

## 2. 使用原则

### 2.1 验收对象不是“页面像不像原型”，而是“是否承接了正确的产品对象和状态行为”

因此每页验收都至少应检查：

- 信息结构是否正确
    
- 关键状态是否覆盖
    
- 操作路径是否可达
    
- 不应展示的内容是否被克制隐藏
    

### 2.2 验收必须覆盖三类层面

- **静态结构验收**：页面区块、组件和字段是否在位
    
- **状态行为验收**：不同状态下是否按规范变化
    
- **联动验收**：多个区块是否同步变化

# 3. 页面级验收清单

## 3.1 Scenes 页面

### 3.1.1 静态结构验收

-  页面能展示 Scene 列表
    
-  每个 SceneCard 至少展示 Scene Name、Description、Analysis Theme、最近任务状态、待处理事项数、更新时间
    
-  页面提供 Create Scene 入口
    
-  页面提供 Open Scene 操作
    
-  页面提供 Duplicate / Archive 入口（如当前版本已实现）
    

### 3.1.2 信息边界验收

-  不把 Skill key 作为主信息展示
    
-  不把 capability key、graph digest 等内部对象放进列表主区
    

### 3.1.3 空状态验收

-  无 Scene 时有明确空状态说明
    
-  空状态下有 Create Scene 引导
    

---

## 3.2 Scene Overview 页面

### 3.2.1 静态结构验收

-  展示 Scene 摘要
    
-  展示最近任务状态摘要
    
-  展示最近结果摘要
    
-  展示待处理事项
    
-  展示小地图预览
    
-  展示最近活动流
    

### 3.2.2 行动入口验收

-  可进入 Workbench
    
-  可查看 Task Runs
    
-  可查看 Results
    

### 3.2.3 信息层级验收

-  该页以摘要为主，而不是细节治理页
    
-  不堆叠过多技术字段
    

---

## 3.3 Scene Workbench 页面

### 3.3.1 结构验收

-  存在三栏结构：左（Inputs/Layers）、中（Map）、右（Analysis/Task/Context）
    
-  地图是视觉中心
    
-  Header 区存在且展示 Scene Name、Analysis Type、Model、Current State、Required Inputs Ready、主按钮
    
-  Inputs Tab 存在
    
-  Layers Tab 存在
    
-  Analysis Tab 存在
    
-  Task Tab 存在
    
-  Context Tab 存在
    

### 3.3.2 输入绑定验收

-  Inputs Tab 区分 Required / Optional / Uploaded but Unbound
    
-  每个输入角色显示 Expected Type、当前绑定状态、动作按钮
    
-  Missing Required Inputs 能被高亮
    
-  Invalid Bindings 能被高亮
    

### 3.3.3 AI 对话区验收

-  用户可通过 AI 发起分析请求
    
-  AI 区能显示 Analysis Type / Model / Suggested Next Steps 等分析摘要
    
-  AI 区不是页面唯一主界面
    
-  AI 区不直接暴露 Selected Skill
    

### 3.3.4 地图区验收

-  地图持续可见 Scene 范围
    
-  图层可切换可见性
    
-  可 Zoom to Layer
    
-  地图状态栏显示 Coordinates / Scale / CRS / Active Layer
    

### 3.3.5 Task / Context 区验收

-  Task Tab 能清晰区分 System Facts 和 Suggestions
    
-  Context Tab 能展示 Scene Facts、Bound Inputs、Execution Context、Manifest Summary
    

### 3.3.6 行为验收

-  用户可上传资产
    
-  用户可绑定/替换输入
    
-  用户可触发 Run Analysis（在 Ready 条件下）
    
-  WAITING_USER 时 Workbench 能继续作为问题处理入口
    

---

## 3.4 Tasks 页面

### 3.4.1 结构验收

-  存在搜索框
    
-  存在状态 Tabs 或等价筛选方式
    
-  列表中展示 Task ID、Scene、Analysis Type、Model、Current State、Updated At、Can Resume、Result Available
    

### 3.4.2 使用性验收

-  用户可以从 Tasks 快速进入 Task Detail
    
-  waiting / running / failed / succeeded 任务可快速筛选
    

---

## 3.5 Task Detail / Governance 页面

### 3.5.1 结构验收

-  存在 Top Summary
    
-  存在 Lifecycle Panel
    
-  存在 Governance Panel
    
-  存在 Events Tab
    
-  存在 Manifest Tab
    
-  存在 Audit Tab（如角色允许）
    
-  存在 Artifacts / Result Bundle 区
    

### 3.5.2 生命周期验收

-  Lifecycle 至少覆盖 CREATED、COGNIZING、PLANNING、VALIDATING、WAITING_USER、RESUMING、QUEUED、RUNNING、RESULT_PROCESSING、ARTIFACT_PROMOTING、SUCCEEDED、FAILED
    
-  当前阶段能够高亮
    
-  失败阶段能够明确显示
    
-  取消发生阶段能明确显示（如支持）
    

### 3.5.3 治理验收

-  WAITING_USER 时 Governance Panel 是核心区
    
-  能展示 missing inputs、invalid bindings、required actions、can_resume、suggested fix
    
-  WAITING_USER 不是一句普通提示语，而是结构化对象投影
    

### 3.5.4 结果验收

-  成功完成后，Artifacts / Result Bundle 区可以展示主输出与下载入口
    
-  失败时可展示失败摘要或日志引用，而不伪装成成功结果
    

---

## 3.6 Assets 页面

### 3.6.1 结构验收

-  存在资产列表
    
-  每个资产至少展示 Asset Name、Type、Format、Status、Used by Scenes、Used by Tasks、Uploaded At
    

### 3.6.2 Asset Detail 验收

-  存在 Metadata 区
    
-  存在 Map Preview
    
-  存在 Usage Relations
    
-  存在 Binding History
    

---

## 3.7 Results / Result Detail 页面

### 3.7.1 Results 列表验收

-  可按 Scene / Analysis Type / Model / Time 筛选（当前版本至少具备基础筛选）
    
-  每个结果项至少展示 Scene、Analysis Type、Model、Time、Result Summary
    

### 3.7.2 Result Detail 验收

-  展示 Result Summary
    
-  展示 Map
    
-  展示 Metrics
    
-  展示 Explanation
    
-  展示导出入口
    
-  展示关联 Task / Scene / Manifest 摘要
    

### 3.7.3 结果消费验收

-  结果页不是纯文本报告页
    
-  地图、指标、解释三者存在联动或至少共存

# 4. 状态行为验收清单

## 4.1 Workbench 状态验收

### Idle / Draft

-  Header 显示默认引导状态
    
-  Analysis Tab 可输入分析请求
    
-  Task Tab 不伪装成已有任务
    

### Understanding

-  Header 显示识别中/理解中
    
-  Analysis Tab 能显示识别出的分析摘要或识别中状态
    
-  用户不能误以为已经 Ready to Run
    

### Waiting for Required Input

-  Header 显示 Waiting for Required Input 或等价状态
    
-  Inputs Tab 高亮缺失项
    
-  Task Tab 显示 Can Resume / Suggested Fix
    
-  用户可继续上传/绑定/修复
    

### Ready to Run

-  Header 显示 Ready to Run
    
-  主按钮为 Run Analysis
    
-  Inputs Tab 中 required roles 已完成绑定
    

### Queued

-  Header 显示 Queued
    
-  本轮输入切为只读快照
    
-  Results 未被提前开放
    

### Running

-  Header 显示 Running
    
-  存在 Current Stage 或等价阶段提示
    
-  Inputs / 参数本轮只读
    
-  Results 不伪装成已完成
    

### Processing Results

-  Header 显示 Processing Results
    
-  Results 页若已可进入，应为 preparing 占位
    
-  不开放最终下载
    

### Completed

-  Header 显示 Completed
    
-  主按钮为 View Results
    
-  地图可加载结果图层
    
-  Result Detail 正式可用
    

### Failed Recoverable

-  Header 显示 Action Required 或等价语义
    
-  Task Tab / Governance Panel 明确 Can Resume
    
-  提供 Fix and Resume 路径
    

### Failed Terminal

-  Header 显示 Failed
    
-  不显示可 Resume
    
-  提供 Start New Analysis 或复制重发路径
    

---

## 4.2 Governance 状态验收

### WAITING_USER

-  Lifecycle 高亮 WAITING_USER
    
-  Governance Panel 是主区
    
-  required actions 清晰可见
    

### RUNNING

-  Lifecycle 高亮 RUNNING
    
-  Events 持续可查看
    
-  Governance 不再把修复面板作为主区
    

### RESULT_PROCESSING

-  Lifecycle 高亮 RESULT_PROCESSING
    
-  Artifacts / Result Bundle 展示 preparing 状态
    

### SUCCEEDED

-  Lifecycle 完成态正确
    
-  Artifacts / Result Bundle 可用
    
-  Report 入口可用
    

### FAILED

-  Lifecycle 明确失败位置
    
-  Governance Panel 清楚展示是否可恢复
    

---

# 5. 跨组件联动验收清单

## 5.1 Workbench 联动

-  用户提交分析请求后，Header / Analysis / Task / Context 同步变化
    
-  识别出 Analysis Type / Model 后，Header 与 Analysis Tab 一致
    
-  绑定输入后，Required Inputs Ready 与 Inputs Tab 状态同步更新
    
-  校验通过后，Header 进入 Ready，Task Tab 显示 Validation Passed
    
-  提交运行后，Header / Task Tab / Inputs Tab 同步切到本轮只读运行态
    
-  结果完成后，Map、Layers、Header、Report Entry 同步更新
    

## 5.2 地图联动

-  点击输入对象时，Inspect Panel 或等价区域能展示对象摘要
    
-  点击结果对象时，能进入结果解释/摘要链路
    
-  Active Layer 切换后，状态栏与 Inspect 上下文同步变化
    

## 5.3 Governance 联动

-  Lifecycle 状态变化时，Top Summary 同步变化
    
-  WAITING_USER 时，Governance Panel 与 Resume 入口同步开放
    
-  成功完成后，Artifacts / Result Bundle / Report Entry 同步可用

# 6. 信息边界验收清单

## 6.1 用户侧不应默认出现的内容

-  Skill
    
-  Selected Skill
    
-  capability key
    
-  graph digest
    
-  internal prompt version
    
-  思维流 / 推理链
    

## 6.2 建议性信息不得伪装成权威事实

-  Suggested Fix 与 Current State 视觉区分明显
    
-  AI 解释不替代结构化 Task Facts
    
-  可 Resume / 可 Cancel / 结果可用性由权威 UI 表达
    

---

# 7. 空状态 / 错误状态验收清单

## 7.1 空状态

-  无 Scene 时有创建引导
    
-  无输入时有上传/绑定引导
    
-  无结果时有合理占位，不留空白页
    
-  无任务时不误导用户认为系统异常
    

## 7.2 错误状态

-  页面加载失败有 Retry
    
-  单个组件失败不拖垮整个页面
    
-  任务失败时展示恢复路径或重新发起路径
    
-  Heartbeat Lost（如支持）被渲染为状态不确定，而非成功/失败

# 8. 角色化验收清单

## 8.1 普通用户

-  可完成完整分析主流程
    
-  不默认看到 Audit 与内部编排信息
    
-  可查看结果与解释
    

## 8.2 高级用户

-  可开启 Advanced View
    
-  可看到更完整 Manifest / Validation / Runtime Summary
    

## 8.3 运维 / 审计用户

-  可查看 Audit Tab
    
-  可查看更完整 Events / Artifacts / Result Bundle 摘要

# 9. 最小链路验收场景

以下场景应至少完整走通一次。

## 场景 A：从对话发起到 WAITING_USER

-  用户在 Analysis Tab 中发起分析请求
    
-  系统识别 Analysis Type / Model
    
-  缺失输入被高亮
    
-  Task Tab 和 Governance 中都能看到待补信息
    

## 场景 B：补输入并恢复到 Ready / Run

-  用户绑定缺失输入
    
-  Required Inputs Ready 更新
    
-  状态退回或进入 Ready to Run
    
-  用户可点击 Run Analysis
    

## 场景 C：Queued → Running → Processing Results → Completed

-  Header 与 Task/Governance 状态逐步变化
    
-  Running 阶段存在阶段提示
    
-  Processing Results 阶段不提前开放结果
    
-  Completed 后 Result Detail 可用
    

## 场景 D：Failed Recoverable

-  失败后显示 Action Required
    
-  可 Resume 路径清晰可达
    
-  修复后可重新推进
    

## 场景 E：Failed Terminal / Cancelled

-  不可 Resume 时 UI 清晰表达
    
-  用户可重新发起或复制配置再分析
    
-  旧轮次不伪装成可继续执行

# 10. 使用建议

## 给设计师

- 用本文检查页面是否真正承接了结构和状态
    
- 不要只看静态原型
    

## 给前端

- 把本文转成自测清单与联调清单
    
- 优先检查状态流转与联动
    

## 给产品 / 架构

- 用本文做版本验收
    
- 用最小链路场景确认系统主闭环是否成立
