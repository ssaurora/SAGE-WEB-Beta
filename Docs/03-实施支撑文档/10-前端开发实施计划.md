## 1. 文档目的

本文用于定义当前系统前端侧的实施路线、阶段目标、开发范围、交付物与验收标准。

本文面向的不是通用 Web 应用，而是一个以 **Scene 为工作容器、以 Workbench 为分析主界面、以 GIS 地图为核心承载、以 Task Governance 为治理闭环、以 Results 为结果消费出口** 的智能分析系统前端。

因此，前端开发的目标不是单纯完成页面绘制，而是承接系统总体架构中的以下关键对象：

- Scene
    
- Task
    
- Input Binding
    
- Map / Layer
    
- Result
    
- Governance / Audit
    
- AI-guided Analysis Interaction
    

前端实施必须服务于以下总体目标：

1. 支撑用户以场景为单位组织分析工作；
    
2. 支撑用户在 Workbench 中完成分析发起、输入绑定、运行观察与结果进入；
    
3. 支撑任务全生命周期的结构化治理与状态可视化；
    
4. 支撑分析结果的地图化、指标化与解释化消费；
    
5. 保证普通用户界面简洁可用，同时为高级模式与审计模式保留受控扩展空间。
    

---

## 2. 实施原则

### 2.1 主链优先原则

前端开发必须优先保证一条完整用户主链成立，而不是平均推进所有页面。

主链定义为：

**进入 Scene → 在 Workbench 发起分析 → 补齐输入 → 观察任务状态推进 → 获取结果 → 查看结果解释 → 必要时进入治理页追踪任务事实。**

### 2.2 GIS-first 原则

地图不是装饰性组件，而是分析界面的核心工作画布。

因此，涉及分析、输入、结果、图层的页面必须围绕地图组织，而不能退化为“聊天页面 + 小地图预览”。

### 2.3 结构化优先原则

系统中的关键治理事实不得仅以自然语言或消息流形式存在，必须通过结构化前端界面承接，包括但不限于：

- 生命周期状态
    
- 运行阶段
    
- 缺失输入
    
- 参数摘要
    
- 工件
    
- 审计信息
    
- 结果摘要
    

### 2.4 用户视角收敛原则

普通用户视角下，前端应围绕“分析任务”和“InVEST 模型能力”组织，不应暴露 Skill 等内部工程对象。

### 2.5 可扩展原则

当前前端虽以 water_yield 为最小真实案例，但其页面框架、状态机制与治理结构必须具备扩展到多模型、多场景、多结果类型的能力。

---

## 3. 开发范围总览

本期前端开发范围建议覆盖以下模块：

1. 全局应用底座
    
2. Scenes 模块
    
3. Scene Detail 容器
    
4. Workbench 分析主界面
    
5. Tasks 与 Task Governance 模块
    
6. Results 与 Result Detail 模块
    
7. 专项规范落地
    
8. 联调与验收收口
    

不建议在本期优先投入大量资源建设以下内容：

- 面向普通用户暴露的 Skill 管理界面
    
- 复杂工作流编排器式页面
    
- 过度细化的系统管理后台
    
- 先于主链闭环的大规模高级模式能力堆叠

## 4. 阶段划分与实施计划

# 4.1 Phase 0：前端基础底座搭建

## 4.1.1 阶段目标

建立后续所有页面开发所依赖的通用底座，统一页面框架、设计系统、状态表示与数据接入方式。

## 4.1.2 开发项

### （1）应用框架与路由骨架

- 全局 Layout
    
- 顶部导航 / 侧边导航
    
- 一级路由与场景级容器路由
    
- 页面容器规范
    

### （2）设计系统基础组件

- Typography 层级
    
- 色板与状态色
    
- Spacing / Radius / Shadow Token
    
- Card
    
- Panel
    
- Tabs
    
- Table
    
- Badge
    
- Drawer
    
- Modal
    
- EmptyState
    
- Skeleton
    
- ErrorState
    

### （3）状态与枚举统一

- Task State 枚举前端映射
    
- Stage State 枚举前端映射
    
- 状态标签、文案、颜色、图标统一
    
- 页面级空态 / 加载态 / 错误态标准化
    

### （4）数据层基础

- 查询封装
    
- 命令调用封装
    
- SSE / 轮询接入预留
    
- ViewModel 适配层
    
- 错误回退机制
    

### （5）模式与权限底座

- 普通模式
    
- 高级模式预留
    
- 审计模式预留
    
- 页面级权限可见性控制预留
    

## 4.1.3 交付物

- 前端项目基础工程
    
- 通用组件库初版
    
- 页面框架模板
    
- 状态映射配置表
    
- 数据获取与更新基础封装
    

## 4.1.4 验收标准

- 任意页面可复用统一组件与状态样式；
    
- 任务状态展示不再由各页面自行定义；
    
- 页面基础容器、弹窗、表格、空态风格统一；
    
- 已具备承接 Scene / Workbench / Governance / Results 的基础能力。
    

## 4.1.5 风险控制

- 避免边做页面边定义组件，导致 UI 风格持续漂移；
    
- 避免各页面自行解释状态枚举，造成状态呈现不一致；
    
- 避免把 SSE、轮询、查询逻辑散落到页面组件内部。

# 4.3 Phase 2：Workbench 主链闭环

## 4.3.1 阶段目标

完成系统最核心的分析工作台，实现用户从分析发起到结果进入的完整操作主链。

## 4.3.2 页面定位

Workbench 应为系统主工作界面，而非普通聊天页面。

推荐采用三栏结构：

- 左侧：Inputs / Layers
    
- 中部：Map Canvas
    
- 右侧：Analysis / Task / Context
    

## 4.3.3 开发项

### （1）Workbench Header

- Scene 名称
    
- 当前分析类型
    
- 当前模型/能力标识
    
- 当前状态
    
- 输入就绪情况
    
- 主操作按钮
    

### （2）Inputs 面板

- Required Inputs
    
- Optional Inputs
    
- Uploaded but Unbound
    
- 每个角色的绑定状态
    
- Bind / Replace / Remove / Inspect 操作
    

### （3）Layers 面板

- Scene Boundary
    
- 输入图层
    
- 中间过程图层（如有）
    
- 结果图层
    
- 图层开关、排序、透明度、图例
    

### （4）Analysis 面板

- AI 对话入口
    
- 用户请求摘要
    
- 当前分析说明
    
- 修复建议摘要
    
- Resume 提示
    

### （5）Task 面板

- 当前任务状态
    
- 当前运行阶段
    
- can_resume
    
- can_cancel
    
- 缺失输入提示
    
- 校验失败提示
    
- 结果可用提示
    

### （6）Context 面板

- 当前分析上下文摘要
    
- 绑定概览
    
- 参数摘要
    
- 只读 Manifest Summary 投影
    

### （7）Map Canvas

- 地图主视图
    
- 输入图层可视化
    
- 结果图层可视化
    
- 图层联动
    
- 对象点击反馈
    
- 缩放、定位、图例基础能力
    

## 4.3.4 必须承接的状态

至少覆盖：

- Draft
    
- Waiting for Required Input
    
- Ready to Run
    
- Queued
    
- Running
    
- Result Processing
    
- Completed
    
- Failed
    
- Cancelled
    

## 4.3.5 交付物

- Workbench 页面正式版
    
- 分析主链交互闭环
    
- 状态驱动的 Workbench UI
    
- 地图与输入/结果联动初版
    

## 4.3.6 验收标准

- 用户能在 Workbench 发起分析；
    
- 缺失输入时界面能明确定位问题并引导绑定；
    
- 运行中状态有明确视觉反馈；
    
- 成功完成后结果入口自动显现；
    
- 失败时能看到结构化失败信息而非仅聊天提示；
    
- 地图始终是分析主界面中心。
    

## 4.3.7 风险控制

- 严禁把 Workbench 退化成聊天窗口；
    
- 严禁把地图缩成附属组件；
    
- 严禁把输入绑定逻辑隐藏在自然语言消息里；
    
- 严禁将任务状态、结果可用性、修复建议仅用文本描述而无结构化 UI 承接。

# 4.4 Phase 3：Tasks 与 Task Governance 页面建设

## 4.4.1 阶段目标

将任务从“可以跑”升级为“可以治理、可以追踪、可以解释”。

## 4.4.2 开发项

### （1）Tasks 页面

- 任务列表
    
- 按状态筛选
    
- 按 Scene 筛选
    
- 最近更新时间
    
- 待处理提示
    

### （2）Task Detail / Governance 页面

建议包含以下区块：

- 顶部任务摘要卡
    
- 生命周期步骤条
    
- 当前治理状态摘要
    
- Events / Timeline
    
- Manifest 只读视图
    
- Artifacts 列表
    
- Result Bundle 摘要
    
- Audit 摘要
    
- 失败原因或恢复提示区
    

### （3）治理相关弹窗与抽屉

- Cancel 确认弹窗
    
- Resume 指导弹窗
    
- 失败详情抽屉
    
- 工件详情抽屉
    
- 审计详情抽屉
    

## 4.4.3 交付物

- Tasks 页面
    
- Task Detail 页面
    
- Governance 相关交互组件
    

## 4.4.4 验收标准

- 任一任务都可从列表进入详情治理页；
    
- 生命周期条与当前状态一致；
    
- waiting / failed / cancelled / completed 均有不同治理表现；
    
- 工件、审计、结果摘要均能结构化查看；
    
- 页面可清晰表达“下一步该做什么”。
    

## 4.4.5 风险控制

- 避免把治理信息继续塞进 Workbench 右侧对话区；
    
- 避免 Task Detail 只有状态字段，没有生命周期、事件流和工件事实；
    
- 避免用户在失败场景下看不出是输入问题、执行问题还是取消问题。

# 4.5 Phase 4：Results 与 Result Detail 建设

## 4.5.1 阶段目标

使前端具备正式承接分析结果的能力，而不是让结果停留在任务结束提示上。

## 4.5.2 开发项

### （1）Results 页面

- 结果列表
    
- 按 Scene / Task 关联查看
    
- 更新时间与状态摘要
    

### （2）Result Detail 页面

- Result Summary
    
- 指标摘要卡
    
- 解释文本区
    
- 地图叠加区
    
- 输入/输出对照区
    
- 结果下载入口
    
- 回跳 Task / Scene 入口
    
- 参数摘要只读区
    

## 4.5.3 交付物

- Results 页面
    
- Result Detail 页面
    
- 结果地图联动与指标展示模块
    

## 4.5.4 验收标准

- 用户可明确知道“本次分析产生了什么结果”；
    
- 指标、解释、地图三者可联合消费；
    
- 结果页面不再依赖任务消息流理解输出；
    
- 用户可以从结果页回溯到场景与任务上下文。
    

## 4.5.5 风险控制

- 避免结果页仅显示下载链接；
    
- 避免结果解释与任务上下文脱节；
    
- 避免地图结果展示与指标区分离，导致结果不可理解。

# 4.6 Phase 5：专项规范落地

## 4.6.1 阶段目标

将当前已有的专项规范从文档层落实为统一前端行为。

## 4.6.2 专项一：页面状态流转与运行状态规范

### 开发内容

- 主状态映射表落地
    
- 阶段状态映射表落地
    
- 各页面状态联动规则实现
    
- heartbeat / running / queued / canceling / result_processing 的细粒度表现
    

### 验收标准

- 同一状态在不同页面表现一致；
    
- running 与 result_processing 有明显区分；
    
- cancel 具有明确过渡反馈。
    

## 4.6.3 专项二：参数可见性规格

### 开发内容

- 参数分层显示
    
- 普通模式 / 高级模式区分
    
- 只读参数与可编辑参数区分
    
- 参数解释入口
    

### 验收标准

- 普通用户不面对底层 schema；
    
- 参数展示围绕分析语义组织，而不是裸字段输出；
    
- 高级模式下可展开更深层参数信息。
    

## 4.6.4 专项三：地图交互规格

### 开发内容

- 图层树交互
    
- 图例规则
    
- 对象点击反馈
    
- 输入图层与结果图层联动
    
- Scene / Workbench / Result Detail 三类地图的差异化行为
    

### 验收标准

- 地图交互服务于分析任务；
    
- 图层信息有秩序，不混乱堆叠；
    
- 结果图层具有明确图例与解释关系。
    

## 4.6.5 专项四：权限与角色规格

### 开发内容

- 普通用户可见范围
    
- 高级用户可见范围
    
- 审计/运维用户可见范围
    
- 页面与操作入口按角色受控开放
    

### 验收标准

- 不同角色看到的是信息深度差异，而不是完全不同的系统；
    
- 审计信息不会在普通视角过度暴露；
    
- 治理与审计入口可控可解释。
    

## 4.6.6 风险控制

- 避免专项规范只写在文档中但未固化到界面行为；
    
- 避免不同页面、不同角色、不同状态下出现相互冲突的展示逻辑。

### （2）全链路走查

至少覆盖以下典型路径：

- 创建 Scene
    
- 发起分析
    
- 缺失输入
    
- 补齐输入
    
- Resume
    
- Running
    
- Completed
    
- 查看 Result
    
- 查看 Governance

### （3）统一验收

- 结构验收
    
- 状态验收
    
- 交互验收
    
- 视觉一致性验收
    
- 术语统一验收
    

## 4.7.3 交付物

- 联调完成版本
    
- 全链路验收清单
    
- 缺陷修复清单
    
- 可演示版本
    

## 4.7.4 验收标准

- 核心主链可稳定演示；

- 避免只做静态页面验收，不做状态行为验收；
    
- 避免不同页面各说各话，导致术语和状态表达冲突。

- 弹窗 / 抽屉 / 表格 / 空态统一
    
- 视觉一致性
    
- 多状态表现规范落实
    

---

## 7. 关键风险与管理建议

### 7.1 最大风险：主链不成立

如果 Workbench 无法完整承接“发起分析—补输入—看状态—拿结果”，则即便页面很多，前端仍未实现系统目标。

### 7.2 最大设计偏差：退化成聊天应用

本系统不是聊天产品。AI 交互必须嵌入结构化工作台，而不能取代结构化界面。

### 7.3 最大可用性风险：地图弱化

地图若不成为分析中心，则整个 GIS-first 定位会失真。

### 7.4 最大治理风险：事实不结构化

生命周期、事件、工件、审计若未结构化承接，系统将无法形成真正可治理闭环。

### 7.5 最大扩展风险：当前页面写死单案例

当前虽围绕 water_yield，但页面结构不能写死为单一案例页面，而应保留对多模型与多结果结构的扩展能力。

---

## 8. 里程碑建议

建议设置以下四个正式里程碑：

### M1：Scene 容器可用

用户可以创建场景、进入场景并查看场景级摘要。

### M2：Workbench 主链成立

用户可以在 Workbench 中发起分析、补齐输入、查看状态并进入结果。

### M3：Task Governance 成立

用户可以查看任务生命周期、事件、工件与审计摘要，并理解下一步动作。

### M4：Result Consumption 成立

用户可以消费结果地图、指标和解释，并可回溯任务与场景上下文。

---

## 9. 结论

当前前端开发不应再以“页面是否足够多”为衡量标准，而应以是否正式承接以下七类对象为标准：

- Scene
    
- Task
    
- Input
    
- Map
    
- Result
    
- Governance
    
- Analysis Interaction
    

因此，推荐实施策略为：

**先搭底座，后打主链；先形成 Workbench 闭环，再补齐治理与结果页面；最后通过专项规范与联调完成收口。**

这一路径能够最大程度保证当前前端建设与系统总体设计保持一致，并为后续多模型、多场景、多结果类型扩展建立稳定基础。
