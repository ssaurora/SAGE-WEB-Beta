# AppShell + 全局导航 + TaskContextBar 详细设计稿

**适用范围**：`SAGE-WEB-Beta` 整体应用层
**设计对象**：`AppShell`、`primaryNav`、Scene 内导航、跨页任务上下文条
**设计目标**：把当前“页面可用、链路已通”的前端，进一步收敛为“具有统一任务叙事的科研任务工作台”

---

## 1. 设计目标

当前前端已经具备较清晰的任务主链路：

- `Overview` 作为状态入口页，已能根据状态决定唯一主动作
- `Workbench` 已经基于 `layoutMode` 切出三种主布局模式，开始由状态改写页面骨架
- `Task Governance` 已完成“决策区前置 + 证据区 tabs 化”的关键收口

但整体应用层仍存在一个明显短板：

> 页面已经开始像任务系统，壳层和导航却还像传统后台。

当前 `AppShell` 仍然使用：

- 左侧功能菜单式导航
- 顶部页面标题式 header

这会持续削弱你已经在单页内部建立起来的任务叙事。

因此，这份设计稿的目标是：

1. 让壳层从“页面容器”升级为“任务上下文容器”
2. 让导航从“功能菜单”升级为“工作入口”
3. 让跨页跳转从“页面切换”升级为“任务连续推进”

---

## 2. 现状问题

## 2.1 AppShell 的问题

当前 `AppShell` 主要特征是：

- 左侧为 `primaryNav`
- 顶部为品牌 + `getPageTitle(pathname)` 生成的页面名
- 主区为普通内容容器

这套结构的问题不是“错”，而是“太弱”。
它只能回答：

- 我在哪个页面

却回答不了：

- 我现在在哪个 Scene / Task 上
- 当前任务处于什么状态
- 我在整个分析链路里的哪个位置

这对于科研任务系统来说是不够的。

---

## 2.2 全局导航的问题

当前 `primaryNav` 仍然是：

- `Scenes`
- `Tasks`
- `Assets`
- `Reports`
- `Settings`

这里至少有三个问题：

### 问题一：顶层对象粒度不一致

`Scenes` 是工作容器，`Tasks` 是过程对象，`Reports` 是结果消费对象，三者不是同一层级。

### 问题二：`Tasks` 削弱了 `Scene-first`

当 `Tasks` 作为一级对象存在时，用户会自然认为：

- 任务可脱离场景独立漫游
- Scene 只是可选上下文，而不是主容器

这和 `Scene-first analysis workbench` 的定位是冲突的。

### 问题三：`Reports` 与 `Results` 语义冲突

Scene 内已经使用 `Results`，全局却仍然使用 `Reports`，这会造成：

- 报告和结果的关系不清
- 结果消费路径不统一

---

## 2.3 页头的问题

当前 header 的右侧只是显示 `getPageTitle(pathname)` 的结果，如：

- `Scene Workbench`
- `Scene Overview`
- `Task Governance`
- `Reports` 等

这是一种典型的后台页头思维。
它对用户的帮助非常有限，因为用户更关心的是：

- 当前任务是什么
- 当前状态是什么
- 现在是继续运行、补输入还是看结果

而不是“这个页面英文叫啥”。

---

## 2.4 缺少统一任务上下文条

虽然各页面已经通过 `from / taskId / contextTaskId` 在逻辑上保留上下文，
但界面层面仍然缺少一个统一、稳定、持续可见的任务上下文条。

结果就是：

- 技术上上下文是连着的
- 视觉上用户还是会觉得自己在跳页面

---

## 3. 重构总原则

## 3.1 Scene-first 必须体现在导航层

只要顶层导航还把 `Tasks` 和 `Scenes` 并列，Scene-first 就只是文案，不是产品结构。

## 3.2 Header 必须优先表达任务上下文

Header 的主要职责不再是显示页面名，而是显示：

- Scene
- Task
- State
- 当前链路位置

## 3.3 全局导航必须只保留长期工作入口

全局导航不应再承载所有对象类型，而应只保留用户长期会反复进入的全局入口。

## 3.4 跨页连续性必须显式可见

任务从 Overview 到 Workbench，再到 Governance 和 Results，必须让用户感受到“这是同一条链”。

---

## 4. AppShell 详细设计

# 4.1 目标定位

新的 `AppShell` 应定位为：

> 轻壳层 + 强上下文头部 + 页面主内容区

它的职责不是塑造页面视觉中心，而是保证：

- 全局入口稳定
- 当前任务上下文稳定
- 页面主区不被壳层抢权

---

# 4.2 新结构

建议将 `AppShell` 统一为三层：

## A. 左侧：Global Rail

仅承载真正的全局入口

## B. 顶部：Context Header

显示当前任务上下文，而不是页面名

## C. 主区：Page Content

页面自己负责表达内容，不由壳层替页面做标题管理

---

# 4.3 Global Rail 设计

## 保留项

建议左侧只保留：

- Scenes
- Assets
- Results
- Settings

其中：

- `Results` 仅在你确认需要全局结果消费入口时保留
- 若没有明确全局结果消费需求，则 Results 也可退为 Scene 内对象

## 删除项

- `Tasks`
- `Reports`

### 说明

`Tasks` 下沉后，任务相关浏览只通过以下几种方式进入：

- Scene 内 `Task Runs`
- Workbench / Governance / Results 上下文入口
- 统一任务搜索入口（若以后真的需要）

`Reports` 必须和 `Results` 统一命名，不允许继续双语义并存。

## 视觉原则

Global Rail 必须继续保持现在的轻量风格，不再恢复为大块卡片式导航。
当前左侧轻导航方向是对的，只是对象命名和分层还不够干净。

---

# 4.4 Context Header 设计

## 当前问题

当前 header 左边是品牌，右边是页面标题。
这太弱，也太后台。

## 新 header 结构建议

### 左侧：品牌弱识别

保留：

- `SAGE-WEB`
- Beta 标识可弱化
- icon 可保留但缩弱

### 中部：Task Context

主要显示：

- 当前 Scene 名称
- 当前 Task ID（若存在）
- 当前状态 badge
- 当前页面在链路中的位置

示例：
`Scene: Gura Water Yield / Task: task-001 / State: Running / Workbench`

### 右侧：全局模式与角色

可保留：

- 当前角色（Analyst / Viewer）
- 模式状态（Read Only / Editable）

不要再只显示 page title。

## 无任务上下文场景

例如在 `Scenes` 首页或 `Settings` 页面，没有 task context 时，header 可退化为：

- 当前全局对象名
- 一句轻量描述

但一旦进入 Scene 主链路页面，header 必须升级成上下文栏。

---

# 4.5 Main Content 容器设计

`<main className="p-6 lg:p-8">` 目前是通用 padding 容器。
这个方向没问题，但要补两个约束：

## 约束一

壳层不再给页面加额外 page title 区。

## 约束二

页面顶部的第一块应优先保留给：

- `TaskContextBar`
- 或页面自己的主判断区

而不是重复的标题区。

---

## 5. 全局导航详细设计

# 5.1 新的 primaryNav

建议直接改为：

```ts
export const primaryNav: NavItem[] = [
  { label: "Scenes", href: "/scenes", description: "场景入口与工作容器" },
  { label: "Assets", href: "/assets", description: "全局资产与绑定素材" },
  { label: "Results", href: "/results", description: "结果消费与报告入口" },
  { label: "Settings", href: "/settings", description: "角色与模式设置" },
];
```

若暂时没有 `/results` 全局页，则先不加。

---

# 5.2 导航语义要求

## Scenes

永远是第一入口。
这是整个产品的主容器。

## Assets

只表示全局资产，不再与 Scene Assets 竞争。

## Results

只表示跨 Scene 的结果消费入口。
如果没有跨 Scene 结果需求，则不应强行设为一级对象。

## Settings

全局系统设置页，可以保留。

---

# 5.3 明确下沉的对象

以下对象一律不应继续作为顶层导航：

- Tasks
- Task Governance
- Task Runs
- Audit
- Scene Assets
- Scene Results

这些都属于：

- 场景内对象
- 任务上下文对象
- 支撑视图对象

---

## 6. Scene 内导航详细设计

# 6.1 当前结构

当前 `sceneNav(sceneId)` 为：

- Overview
- Workbench
- Assets
- Task Runs
- Results
- Audit

这套结构已经具备基础，但需要**显式分层**。

---

# 6.2 新分层建议

## 主链路导航

- Overview
- Workbench
- Results

## 支撑导航

- Assets
- Task Runs
- Audit

---

# 6.3 呈现方式

## 主链路

建议显示为：

- 明显 tab
- 或 header 下方一级横向导航

## 支撑导航

建议显示为：

- “More” 下拉
- 次级 tabs
- 或 secondary nav

目标是让用户一眼看出：

- 哪三个是任务推进页面
- 哪三个是辅助信息页面

---

## 7. TaskContextBar 详细设计

这是整体层最关键的新组件。

# 7.1 组件定位

`TaskContextBar` 是一个横向上下文条，放在页面主内容最上方，用于表达：

- 当前 Scene
- 当前 Task
- 当前状态
- 当前链路位置
- 可选的来源上下文

它不是 breadcrumb，不是普通页头，而是：

> 任务连续性的可视化锚点

---

# 7.2 适用页面

建议在以下页面统一使用：

- `Scene Overview`
- `Scene Workbench`
- `Task Governance`
- `Scene Results`
- `Result Detail`

其中：

- `Scenes` 首页不用
- `Settings` 不用
- `Assets` 可选，取决于是否处于 Scene 内上下文

---

# 7.3 字段规范

## 必填字段

- `sceneName`
- `currentView`（Overview / Workbench / Governance / Results）
- `stateLabel`

## 条件字段

- `taskId`
- `from`
- `role`
- `mode`

## 推荐展示顺序

`Scene / Task / State / View`

例如：

- `Scene: Gura Water Yield`
- `Task: task-001`
- `State: Running`
- `View: Workbench`

---

# 7.4 视觉设计

## 风格

- 低高度
- 弱背景
- 轻边框
- 不抢页面主区

## 语义元素

- Scene 名称可用普通文本
- Task ID 可用 outline badge
- State 用状态 badge
- View 用弱标签或文本

## 行为

- Scene 名称可链接回 Overview
- Task ID 可链接到 Governance
- View 不可点击，仅表示当前位置

---

# 7.5 示例结构

```tsx
<TaskContextBar
  sceneName="Gura Water Yield"
  sceneHref="/scenes/scene-001/overview"
  taskId="task-001"
  taskHref="/task-governance/task-001"
  stateLabel="Running"
  stateVariant="outline"
  currentView="Workbench"
  roleLabel="Analyst"
/>
```

## 8. 页面接入规则

# 8.1 Overview

当前 `Overview` 已经是状态入口页，适合最先接入。
接入后，顶部结构应变为：

1. `TaskContextBar`
2. 当前状态与主动作区
3. Context Evidence 等内容

---

# 8.2 Workbench

当前 Workbench 顶部已有 `Current State` 卡，但缺一个跨页稳定上下文层。
接入后：

1. `TaskContextBar`
2. `Current State`
3. 三种 layoutMode 页面主体

---

# 8.3 Governance

当前 Governance 页顶部已经是决策区，适合在其上方增加 `TaskContextBar`。
这样用户不会感觉自己是“跳到了另一个页面”，而是“进入了当前任务的治理视图”。

---

# 8.4 Results / Result Detail

结果页是最需要上下文连续性的页面。
因为结果消费往往最容易让用户忘记“这是哪个任务跑出来的”。

---

# Phase E：整体走查

- 检查壳层是否仍有后台气质残留

## 11. 最终判断

你现在最缺的，不是再给某一页加一个更好的区块，
而是让整个应用具备这种感觉：

> **“我一旦进入一个 Scene，就像进入了一个连续的分析工作空间。”**

而要做到这一点，最关键的不是单页内部，
而是：

- `AppShell`
- 全局导航
- `TaskContextBar`

这三个一旦落下来，你的前端就会从“页面设计不错”跃迁成“系统感很强”。

- `AppShell` 已从页面标题主导改为上下文态头部，右侧显示区域、角色与模式。

- 主链路与主要支撑页跳转已统一使用共享 `from` 常量，页面内已无散写 `from=xxx` 字面量。

- 全局 `Results` 已正式迁移到 `/results`，旧 `/reports` 保留兼容重定向。
- `Results` 相关页面已统一可见文案，避免 `Reports` / `Results` 双语义混用。
- `Overview`、`Workbench`、`Task Governance` 继续保持“先决策、后证据”的页面顺序。
