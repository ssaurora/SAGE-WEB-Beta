下面直接给你一版**线框级原型描述稿**。
你可以把它当成三张页面的低保真原型说明，交给前端、产品，或者直接喂给 AI 画图工具。

我按你刚才强调的标准来写：

**不是所有内容都塞一页，而是每个模块内部都不要依赖整页上下长滚动。**

# 1. Workbench 线框级原型描述稿

## 1.1 页面定位

Workbench 是**分析主操作界面**。
它不是介绍页，不是报告页，也不是状态汇总页。

用户进入这里，要在一个稳定视口内完成四件事：

* 看当前任务状态
* 看地图上的输入/结果
* 做输入绑定与图层操作
* 看对象详情与下一步动作

---

## 1.2 页面骨架

### 整体结构

采用：

* 顶部固定状态栏
* 中部三栏主体区
* 底部可展开抽屉

页面高度固定为一个完整视口，不允许依赖整页下滚。

---

## 1.3 顶部固定状态栏

### 高度

64px–72px

### 左侧内容

* Scene 名称
* Analysis Type
* Model Name

示例：
`scene-001 · Watershed analysis`
`Water Yield`
`InVEST water_yield`

### 中间内容

* 当前状态 Badge
* Required Inputs Ready
* 当前主提示文本

示例：

* `Waiting for Required Input`
* `3 / 5 required ready`
* `缺少 2 个必需输入，当前不可运行`

### 右侧内容

主操作按钮区，只保留 1~2 个最关键动作：

* Run
* Resume
* Open Governance
* View Results

按钮根据状态切换，不同时出现太多。

---

## 1.4 主体区布局

### 布局比例

* 左栏：320px
* 中栏：自适应主区
* 右栏：360px

### 视觉要求

* 三栏高度一致
* 栏内局部滚动
* 整页不滚

---

## 1.5 左栏：Inputs / Layers 面板

左栏顶部是 Tab 切换：

* Inputs
* Layers

---

### Tab A：Inputs

#### 顶部摘要区

显示三项指标：

* Required Ready
* Missing
* Invalid

示例：

* Ready 3/5
* Missing 2
* Invalid 1

#### 中部：Required Inputs 列表

每项显示：

* 输入名称
* 预期类型
* 当前状态
* 已绑定资源名
* 快速操作按钮

示例条目：

* `Study Area Boundary`

  * Type: Vector Polygon
  * Status: Bound
  * Asset: watershed_boundary.geojson

* `LULC Raster`

  * Type: Raster
  * Status: Missing
  * Action: Bind / Upload

#### 下部：Optional Inputs

结构与 Required 相同，但视觉权重降低。

#### 底部：Upload 区

小型上传入口，不要做得太大：

* 文件名输入/显示
* 资源类型选择
* Upload 按钮

---

### Tab B：Layers

#### 顶部摘要区

* Visible layers count
* Active layer

#### 中部：Layer 分组列表

分为：

* Input Layers
* Result Layers

每个图层项显示：

* 图层名
* Visible / Hidden
* opacity slider
* zoom to layer
* move up/down

#### 视觉规则

这里不要再出现大段说明文本，它是纯操作区。

---

## 1.6 中栏：Map 主画布

### 中栏顶部：悬浮工具条

包含：

* Reset View
* Visible Layers
* Active Layer
* Toggle Input Layers
* Toggle Result Layers
* Fullscreen

### 中栏主体：地图

地图占中栏绝大部分空间，是页面主视觉中心。

### 中栏底部：地图状态条

显示：

* Lng / Lat
* Zoom
* Scale
* CRS
* Active Layer

---

## 1.7 右栏：Inspector / Task / Guidance

右栏顶部是三个 Tab：

* Inspector
* Task
* Guidance

---

### Tab A：Inspector

用于查看地图点击对象详情。

#### 未选中状态

显示一句说明：
`点击地图对象后查看结构化属性。`

#### 选中后显示

* Object Name
* Object Type
* Status
* Updated At
* Coordinate
* Related Task
* Related Result

底部提供两个链接按钮：

* Open Governance
* Open Result Detail

---

### Tab B：Task

用于看当前任务摘要，不是看全量证据。

显示：

* Current State
* Runtime Progress
* Current Blocking Issue
* Lifecycle Summary
* Quick Link to Governance

如果任务正在运行，突出进度条。
如果任务阻塞，突出阻塞原因。
如果任务完成，突出“View Results”。

---

### Tab C：Guidance

用于承载建议与下一步动作。

显示：

* Suggested Next Steps
* Why this step matters
* Recommended Fix
* 一键动作入口

不要超过 5 条建议。
不要把它写成大段说明文章。

---

## 1.8 底部抽屉

底部抽屉默认收起，展开后占页面下半部分。

Tab 建议为：

* Execution
* Audit
* Manifest

### Execution

* Lifecycle events
* Runtime context
* stage summary

### Audit

* 最新审计记录
* 警告/错误摘要

### Manifest

* 只读 manifest 摘要
* 关键参数与输入冻结信息

---

## 1.9 Workbench 原型提示词

你可以直接拿去生成原型图：

**提示词：**
设计一个科研分析系统的 Workbench 页面，面向生态/GIS 分析场景。页面采用固定高度的三栏布局，无整页纵向滚动。顶部是固定状态栏，显示 scene 名称、analysis type、model name、task state、required inputs ready 和主操作按钮。左栏宽约 320px，包含 Inputs 和 Layers 两个 tab，支持输入绑定、上传资源、图层可见性、opacity、zoom to layer。中间是大型 Map 画布，为页面视觉中心，顶部带悬浮工具条，底部带坐标、比例尺、CRS、active layer 状态条。右栏宽约 360px，包含 Inspector、Task、Guidance 三个 tab，用于对象属性、任务状态和下一步建议。底部有可展开抽屉，包含 Execution、Audit、Manifest 三个 tab。整体风格克制、专业、面向科研用户，避免花哨和营销式视觉。

# 2. Governance 线框级原型描述稿

## 2.1 页面定位

Governance 是**任务治理控制台**。
它不是普通详情页，也不是日志长文页。

用户进入这里，只需要迅速回答四个问题：

* 为什么停在这里
* 现在能不能恢复
* 应该做什么
* 证据在哪里

---

## 2.2 页面骨架

### 整体结构

* 顶部固定决策栏
* 中部左右双栏
* 可选底部证据扩展抽屉

---

## 2.3 顶部固定决策栏

### 左侧

* Task ID
* Scene ID

### 中间

* Current State
* Stage State
* Can Resume / Can Cancel / Result Available

### 右侧

主操作按钮：

* Fix and Resume
* Cancel Task
* Re-run
* Back to Workbench

根据状态只高亮一个主动作。

---

## 2.4 主体区布局

### 比例

* 左栏：420px
* 右栏：自适应

### 规则

* 双栏都允许内部滚动
* 整页不滚

---

## 2.5 左栏：Recovery Console

左栏从上到下分四块，但它们都在同一栏里滚，不延长整个页面。

---

### 区块 A：Decision Summary

显示：

* Decision Status
* Decision Outcome
* Next Action Text

示例：

* `Failed Recoverable`
* `优先恢复`
* `优先修复输入问题后执行 Fix and Resume`

---

### 区块 B：Blocking Issues

显示：

* Missing Required Inputs
* Invalid Bindings
* Failure Summary

每项采用清单式，不要大段文字。

---

### 区块 C：Suggested Fix

显示：

* Suggested Fix 列表
* 推荐优先修复项
* 修复原因简述

例如：

* 优先检查 LULC Raster 是否缺失
* 若已上传，检查 role 绑定是否完成
* 若仍失败，核对 raster CRS 与 NoData

---

### 区块 D：Action Area

大按钮区：

* Fix and Resume
* Cancel Task
* Open Workbench
* Open Result (如果有结果)

---

## 2.6 右栏：Evidence Console

右栏顶部是 Tab：

* Evidence
* Manifest
* Lifecycle
* Audit

---

### Tab A：Evidence

显示：

* Failure Summary
* Missing Input Evidence
* Invalid Binding Evidence
* Required Actions
* Suggested Fix Rationale

这是“为什么我建议你这样做”的区域。

---

### Tab B：Manifest

显示：

* Analysis Type
* Model Name
* Required Inputs Ready
* Runtime Profile
* 当前冻结的输入/参数摘要

这里不要展示巨量字段，先显示摘要，必要时可展开。

---

### Tab C：Lifecycle

显示：

* 生命周期事件时间线
* 当前停在哪个阶段
* 最近一次状态迁移
* 是否存在 resume 点

视觉上应采用 timeline，而不是纯文本段落。

---

### Tab D：Audit

显示：

* 审计摘要
* 相关 artifacts
* 错误日志索引
* 下载或查看入口

---

## 2.7 Governance 原型提示词

**提示词：**
设计一个科研分析系统的 Governance 页面，定位为任务治理控制台。页面采用固定高度的双栏布局，无整页纵向滚动。顶部是固定决策栏，显示 task id、scene id、current state、stage state、can resume、can cancel、result available 和主操作按钮。左栏宽约 420px，作为 Recovery Console，包含 Decision Summary、Blocking Issues、Suggested Fix、Action Area 四个区域。右栏是 Evidence Console，顶部带 tab，包含 Evidence、Manifest、Lifecycle、Audit 四个 tab。Evidence 展示失败原因、缺失输入、无效绑定和建议修复依据；Manifest 展示冻结输入与参数摘要；Lifecycle 使用时间线样式展示任务阶段迁移；Audit 展示审计摘要和 artifacts。整体风格严谨、克制、专业，强调治理与恢复，不使用长页面堆叠卡片。

# 3. Results 线框级原型描述稿

## 3.1 页面定位

Results 是**结果查看与解释工作台**。
它不是下载页，也不是报告全文页。

用户进入这里要快速完成：

* 确认当前最值得看的结果
* 查看结果地图或核心指标
* 查看解释与输入输出关系
* 决定继续消费还是回退治理/重跑

---

## 3.2 页面骨架

### 整体结构

* 顶部固定结果状态条
* 中部三栏布局
* 可选底部扩展区

---

## 3.3 顶部固定结果状态条

### 左侧

* Scene ID
* Latest State

### 中间

* Recommended Result
* Decision Hint

### 右侧

主操作按钮：

* Open Recommended Result
* Back to Governance
* Back to Workbench

---

## 3.4 主体区布局

### 比例

* 左栏：320px
* 中栏：自适应
* 右栏：360px

---

## 3.5 左栏：Results List

左栏是结果列表，不要铺到整页正文里。

### 顶部

* 搜索框
* task 过滤
* ready 状态过滤

### 列表项显示

每个结果显示：

* Result ID
* From Task ID
* Generated At
* Explanation Ready
* Map Layer Ready
* Summary 一句话

推荐结果要有明显高亮。

---

## 3.6 中栏：Result Main View

中栏始终展示“当前选中结果”的主视图。

### 如果 map layer ready

显示：

* 结果地图
* 三到五个核心指标
* 一段简短解释摘要

### 如果 map layer not ready

显示：

* 占位提示
* 未就绪原因
* 返回治理或工作台的 CTA

### 如果 explanation not ready

显示：

* 原始指标
* mapping 摘要
* “解释尚未生成”的提示

---

## 3.7 右栏：Result Details

右栏用 Tab：

* Explanation
* Mapping
* Artifacts
* Related Task

---

### Tab A：Explanation

显示：

* 完整结果解释
* 结论摘要
* analyst note / system interpretation

---

### Tab B：Mapping

显示：

* Input → Output mapping
* Source data
* key transformation relationship

建议用双列表或表格，不要用长段文字。

---

### Tab C：Artifacts

显示：

* 可下载文件
* readiness 状态
* artifact type
* 下载按钮

---

### Tab D：Related Task

显示：

* From Task ID
* Task State
* Open Governance
* Open Workbench

---

## 3.8 Results 原型提示词

**提示词：**
设计一个科研分析系统的 Results 页面，定位为结果查看与解释工作台。页面采用固定高度的三栏布局，无整页纵向滚动。顶部是固定结果状态条，显示 scene id、latest state、recommended result、decision hint 和主操作按钮。左栏宽约 320px，为结果列表，支持搜索、task 过滤、ready 状态过滤，每个结果项显示 result id、from task、generated time、explanation ready、map layer ready、summary。中栏为当前选中结果的主视图，优先展示结果地图、关键指标和简短解释摘要；如果结果图层或解释未就绪，则显示占位提示和返回治理/工作台的按钮。右栏宽约 360px，包含 Explanation、Mapping、Artifacts、Related Task 四个 tab。整体风格专业、理性、面向科研用户，强调结果消费和解释，不采用长页面堆叠结构。

# 4. 三个页面统一的样式与交互约束

为了让三页看起来像同一系统，而不是三套不同东西，建议统一以下规则。

## 4.1 统一骨架

三页都统一为：

* 顶部固定局部状态栏
* 中部固定高主体区
* 左/中/右或左/右分栏
* 底部抽屉承载长信息

## 4.2 统一滚动规则

只允许：

* 左栏滚
* 右栏滚
* 抽屉滚

禁止用户为了找核心信息去滚整个页面。

## 4.3 统一按钮逻辑

每页只有一个主按钮高亮。
不要一个页面上同时出现 4 个同权重主动作。

## 4.4 统一信息层级

* 一级：状态、主视图、主动作
* 二级：对象详情、建议、列表
* 三级：日志、manifest、audit、长解释

## 4.5 统一视觉气质

* 不要炫技式大渐变
* 不要过多玻璃态
* 不要营销风
* 用稳定、严谨、专业的科研系统风格

# 5. 给前端或设计师的交付说法

你可以直接这样发给他们：

> 当前系统不需要重做功能语义，但需要重做模块内布局。Workbench、Governance、Results 三个核心模块必须从纵向长滚动页面改为固定视口的工作台布局。原则是：首屏闭环、局部滚动、主次分层、状态变化不改变页面骨架。地图、治理控制、结果主视图分别成为各模块中心，次级信息统一进入侧栏、tab 或底部抽屉，不再通过整页堆叠卡片承载。
