像现在的 workbench、governance、results，都是典型的“从上往下读”的页面，而不是“在一个稳定视口内操作”的页面。`Workbench` 里连续堆了 `Current State / Inputs / Map Canvas / Layers / Next Steps / Execution Context / Object Inspector`，这天然就会把页面拉成长卷。
`Task Governance` 也是先决策区、再恢复区、再证据 tabs，还是纵向堆叠。
`Results` 也先 decision zone，再 context notes，再 results panel，本质还是纵向阅读页。

所以现在该改的，不是“所有东西并到一个页面”，而是：

## 你的每个核心模块，都要从“长页面”改成“视口型页面”

也就是一个模块进入后，用户在首屏里就能同时看到：

* 当前状态
* 核心操作区
* 关键上下文
* 次级信息入口

而不是靠滚动去找。

---

## 我对你现在问题的重新定性

你现在的 UI 不合理，核心不在“信息太多”，而在于三点：

### 1. 模块内部没有稳定骨架

每个模块都像是一组卡片从上往下拼接，而不是一个固定布局的工作界面。

这会导致：

* 首屏重点不稳定
* 不同状态下视觉重心不断漂移
* 用户必须滚动才能拼出完整认知

### 2. 主次关系不清

地图、输入、状态、治理、结果说明，全都在争首屏位置。
结果就是谁都重要，谁都不突出。

### 3. 页面在“阅读”，而不是“操作”

科研系统、分析系统、治理系统，本质都应该是操作界面。
但你现在很多模块更像“项目展示页”或“报告页”。

## 所以要不要整体重新设计 UI？

我的答案是：

**要，但是重设计“模块布局系统”，不是重设计“产品功能结构”。**

也就是：

* 你现在想表达的内容，大体可以保留
* 但每个模块的排版方式，基本都该重做

# 我建议你采用的原则

## 原则一：一个模块只允许一个主操作区

比如：

### Workbench

主操作区只能是地图。

输入、图层、状态、对象详情，都应该围绕地图，而不是和地图平级往下堆。

### Governance

主操作区只能是“恢复 / 判断 / 证据查看”。

不能又是摘要卡、又是说明卡、又是事件列表、又是清单式堆叠。

### Results

主操作区只能是“结果查看与解释”。

不能先讲一堆 decision，再讲 context，再往下才是结果内容。

---

## 原则二：一个模块首屏必须闭环

用户进入模块后，不滚动页面，也应该能完成最核心认知。

例如：

### Workbench 首屏必须回答

* 当前任务状态是什么
* 地图上有哪些东西
* 缺什么输入
* 下一步操作是什么

### Governance 首屏必须回答

* 为什么停在这里
* 能不能恢复
* 应该点什么

### Results 首屏必须回答

* 当前最值得看的结果是哪一个
* 结果是否完整
* 可以继续消费还是要回退处理

如果这些都要靠滚动才能知道，这个模块就失败了。

---

## 原则三：次级信息不要往下堆，要收进侧栏、Tab、抽屉、折叠区

这是你现在最该改的地方。

你现在大量内容是“合理的内容，被错误地用纵向堆叠展示”。

应该改成下面这种方式：

### 适合常驻首屏的

* 状态条
* 主画布 / 主列表 / 主证据区
* 主操作按钮
* 1~2 个关键摘要

### 不适合常驻首屏的

* 生命周期全文
* 审计详情
* Manifest 详细字段
* 扩展解释
* 次级建议
* 对象明细全文

这些都应该进：

* 右侧详情栏
* 下方抽屉
* Tab
* Modal
* Accordion

而不是继续往页面下面接。

# 按你现在这个仓库，我给你具体建议

## 一、Workbench 要重做成“三栏定高布局”

你现在 workbench 最大的问题，是地图虽然重要，但它没有统治页面。 

建议改成：

### 顶部固定状态条

高度固定，永远显示：

* 场景名
* 模型名
* 任务状态
* 必需输入完成度
* 主操作按钮

### 中部主体区：左右中三栏

* 左栏：输入与图层
* 中栏：地图
* 右栏：对象详情 / 状态建议 / 当前任务摘要

### 底部可展开抽屉

* Execution Context
* Next Steps
* Lifecycle
* Audit Snippets

这样用户不需要往下滚动整页。
只需要在局部面板里切换。

### 你现在哪些内容该移动

`Inputs`、`Layers` 不要继续做成地图下方大卡片，应收进左栏。
`Object Inspector` 不要放页面底部，应收进右栏。
`Execution Context` 和 `Next Steps` 不该占主页面，应收进底部抽屉。
`Current State` 不该单独占一整块卡片，应并入顶部状态条。
这些都来自你当前 workbench 组件的内容分布。

---

## 二、Governance 不要再做长页面

你现在 governance 页面的问题，不是信息错，而是纵向串得太长。

建议改成：

### 顶部：Decision Header

只保留：

* 当前任务
* 当前状态
* 是否可恢复
* Primary Action

### 左侧：恢复操作区

* Missing inputs
* Invalid bindings
* Suggested fix
* Resume / Cancel / Re-run

### 右侧：证据区

Tab 切：

* Evidence
* Manifest
* Lifecycle
* Audit

这样 governance 会从“阅读页”变成“治理控制台”。

---

## 三、Results 页面也不要再纵向堆

你现在 results 页仍然有“先说一堆，再给结果”的倾向。

建议改成：

### 左侧：结果列表

列表可筛选、可切换、可标记推荐项

### 中部：当前结果主视图

* 地图结果层 / 指标摘要 / 关键解释

### 右侧：结果详情

Tab 切：

* Explanation
* Inputs→Outputs Mapping
* Downloadables
* Related Task

这样用户不用往下翻着看结果，而是一直在当前视口里切换。

# 你现在最该遵守的版式规则

我建议你后续所有模块统一遵守下面这几条：

## 1. 页面首屏高度必须被设计

不要默认让内容自然往下流。
每个核心模块都要按 `100vh - header` 设计主体区。

## 2. 整页只允许一个纵向滚动容器，最好没有

能局部滚动就局部滚动：

* 左栏滚
* 右栏滚
* 下抽屉滚
  不要整页滚。

## 3. 卡片不是越多越好

现在你有点过度依赖 card 了。
Card 适合承载独立对象，不适合承载整个操作流。

## 4. 固定主视觉中心

Workbench 固定地图为主视觉。
Governance 固定恢复区 + 证据区。
Results 固定结果主视图。
不要每个状态都让主体布局大改。

## 5. 状态变化改内容，不改骨架

你现在 `Workbench` 里不同状态会切不同布局模式，这个方向有一定合理性，但不该让整个页面骨架跟着剧烈变化。
更好的方式是：

* 保持框架不变
* 只替换右栏内容、按钮、提示、局部区域权重
