## 0. 文档定位

  

本文用于定义面向生态问题智能分析系统的用户侧前端设计要求，可直接供设计、前端与产品协同使用。

  

本稿不再讨论底层架构原理，而是将既有总体设计中的正式对象，压缩为用户可见页面、核心区块、显示内容、交互动作与不可暴露对象的统一规范。

  

本文默认以下前提：

  

- 用户通过 **AI 对话** 表达分析意图、补充约束、追问状态、请求解释。

- 系统通过 **场景工作台** 承载地图、输入绑定、任务治理、运行过程与结果查看。

- 用户侧前端 **不直接暴露 Skill**；Skill 仍为内部认知与编排对象。

  

---

  

## 1. 全局产品定位

  

### 1.1 产品形态

  

本系统不是纯聊天产品，也不是纯 GIS 工具台，而是 **AI 引导的场景化分析工作台**。

  

更准确地说，产品应采用以下协同模式：

  

- **AI 对话层**：负责自然语言入口、任务理解反馈、疑问解答、状态解释、修复建议与结果说明。

- **任务工作台层**：负责地图、输入、绑定、运行、治理、结果与审计对象的正式承载。

  

因此，前端设计必须同时满足两点：

  

1. 用户可以随时通过 AI 说“我要做什么”以及“我还有什么疑问”。

2. 地图、输入、任务状态、运行控制、结果与审计不能被聊天流替代。

  

### 1.2 用户侧主概念

  

用户侧应以以下概念组织产品心智：

  

- Scene

- Analysis Type

- Model

- Current State

- Required Inputs

- Bound Inputs

- Suggested Fix

- Result Explanation

- Manifest Status

- Runtime Checks

  

### 1.3 用户侧不作为主概念呈现

  

以下概念默认不作为用户主界面术语出现：

  

- Skill

- Selected Skill

- Skill Route

- Capability Key

- Graph Digest

- Prompt Version

- Runtime Assertions Count

  

这些对象仅可在高级模式、审计模式或调试模式中受控投影。

## 2. 全局设计原则

  

### 2.1 Scene-first

  

系统应以 Scene 作为用户工作的基本容器。用户围绕具体场景组织分析、资产、任务、结果和审计，而不是围绕零散聊天线程工作。

  

### 2.2 Chat-guided

  

AI 对话必须长期存在，并作为分析入口、持续答疑入口和解释入口。但对话区是工作台组成部分，而不是唯一主界面。

  

### 2.3 GIS-first

  

地图是工作台主画布。对于空间输入检查、图层查看、结果叠加和空间判断，地图必须优先于纯文字承载。

  

### 2.4 Governance-visible

  

等待补充、恢复执行、运行中状态、失败原因、Manifest 状态、结果包与审计痕迹必须被结构化呈现，不能退化为零散助手消息。

  

### 2.5 Structured-over-Chat

  

凡是本质上属于结构化对象的内容，应以面板、表单、状态块、列表、图层控制或结果摘要呈现，而不是只存在于聊天记录中。

  

### 2.6 Internal-Complexity Hidden

  

内部的 Skill、受控路由、规划修复、执行图等复杂对象，应经过用户化投影后再展示，避免把前端做成内部开发控制台。

## 7. Scene Workbench（核心页）

  

### 7.1 页面定位

  

场景内主工作模式，是用户进行分析协作的核心页面。

  

### 7.2 页面目标

  

让用户在一个页面里完成：

  

- 通过 AI 对话发起分析请求

- 查看系统当前理解出的分析任务

- 检查输入是否齐备

- 在地图上检查空间对象与图层

- 处理 waiting / resume

- 感知当前任务状态

- 在分析前后继续向 AI 追问与澄清

  

### 7.3 布局要求

  

采用固定三栏：

  

- 左：Inputs / Layers

- 中：Map

- 右：Analysis / Task / Context

  

### 7.4 视觉原则

  

- GIS-first

- 地图主导

- pane / panel 风格

- 高信息密度

- 以场景工作台为主承载，以 AI 对话为主入口、辅助手段和解释界面
