## 1. 文档目的

本文在既有《总体设计》前端章节基础上，进一步输出两类可直接用于实施的交付物：

1. **表格版页面需求总表**：用于设计、前端、产品、架构之间统一页面目标、信息结构与对象边界。
    
2. **组件级规格**：用于前端实现阶段明确组件职责、输入输出、状态、交互与显示边界。
    

本文默认面向你的生态智能分析应用，遵循以下统一前提：

- 用户通过 **AI 对话** 发起分析、追问问题、请求解释。
    
- 用户通过 **Scene Workbench / Task Governance / Result View** 完成正式操作。
    
- 用户侧前端 **不暴露 Skill 作为主概念**。
    
- 前端采用 **GIS-first、Scene-first、Task-governed** 的应用形态。

## 2. 全局设计前提

### 2.1 用户侧主概念

用户侧统一使用以下概念：

- Scene
    
- Analysis Type
    
- Model
    
- Required Inputs
    
- Current State
    
- Missing Required Input
    
- Bound Inputs
    
- Suggested Fix
    
- What’s Needed to Resume
    
- Manifest Status
    
- Runtime Checks
    
- Result Explanation
    

### 2.2 不作为主概念直接暴露

以下概念默认不在普通用户主界面中作为一等概念展示：

- Skill
    
- Selected Skill
    
- Skill Route
    
- Capability Key
    
- Graph Digest
    
- Runtime Assertions Count
    
- Internal Prompt Version
    

这些对象仅可在高级模式、调试模式或审计模式中以受控方式出现。

### 2.3 前端承载原则

- **AI 对话**：负责需求表达、问题澄清、状态解释、结果解释和辅助决策。
    
- **Workbench**：负责地图、输入绑定、分析上下文、任务状态、恢复操作和结果查看。
    
- **Governance 页面**：负责生命周期、事件流、Manifest、Audit、Artifacts、Result Bundle。
    
- **Results 页面**：负责结构化结果消费、地图回看、指标浏览、解释与导出。

# 5. 通用交互规则

## 5.1 AI 对话与正式操作分离

- AI 对话区负责：表达需求、追问、解释、辅助决策。
    
- 正式操作区负责：上传、绑定、运行、恢复、查看结果。
    
- 不允许仅通过聊天消息隐藏式完成关键治理操作。
    

## 5.2 事实与建议分层

- **System Facts**：权威事实，结构化、可审计。
    
- **Suggestions**：辅助说明，可由 AI 生成，但不得伪装为权威状态。
    

## 5.3 高级模式边界

高级模式可以受控显示：

- Workflow / Template
    
- Planning Revision
    
- Manifest advanced details
    
- Runtime Checks 细项
    
- Artifacts advanced metadata
    

但普通模式不应把这些对象前置为核心界面概念。

## 5.4 GIS-first 约束

- 地图在 Workbench 中必须始终是中心区域。
    
- 图层、输入、结果都应服务于空间判断。
    
- 避免把应用做成纯聊天界面或纯表单界面。

# 6. 页面状态流转表

本文档中的“页面状态流转”只定义**前端页面在不同任务/运行状态下的显示与交互变化**，不假设具体后端字段名与接口实现。

设计目标是回答四个问题：

1. 某个状态下，页面应该重点显示什么？
    
2. 某个状态下，用户能做什么、不能做什么？
    
3. 某个状态切换时，哪些页面区域应该联动变化？
    
4. 哪些状态需要从 Workbench 跳转到 Governance，哪些只需局部提示？
    

---

## 6.1 全局状态流转原则

### 6.1.1 状态展示分层

所有页面状态变化统一分成三层：

- **主状态（Primary State）**：用户当前最需要知道的任务/运行状态。
    
- **阶段状态（Stage State）**：任务所处的流程阶段，例如 validating、running、result_processing。
    
- **局部状态（Local UI State）**：页面中某个局部组件的状态，例如输入面板缺项、地图尚未加载、AI 回复中。
    

前端必须避免把大量底层状态直接平铺给用户，而应先归并成用户可理解的页面主状态。

### 6.1.2 页面优先级原则

不同状态下，页面关注点必须变化：

- **待输入 / 待修复**：优先显示 Inputs、Task Facts、Suggested Fix。
    
- **运行中**：优先显示 Current State、Lifecycle、Progress、Cancel 能力、结果占位。
    
- **结果处理中**：优先显示结果准备中、产物整理中、解释生成中。
    
- **已完成**：优先显示结果地图、指标、解释、导出入口。
    
- **失败**：优先显示失败位置、失败原因、可恢复性、下一步建议。
    

### 6.1.3 页面联动原则

同一任务状态变化时，以下区域应联动更新：

- Workbench Header
    
- Analysis Tab
    
- Task Tab
    
- Context Tab
    
- Task Detail / Governance 页的 Top Summary 与 Lifecycle
    
- Results / Results 的可见性与入口状态

## 6.2 全局任务主状态流转表

|主状态|用户感知语义|典型阶段|应重点展示|允许操作|限制操作|推荐页面焦点|
|---|---|---|---|---|---|---|
|Draft / Not Ready|场景已存在，但分析尚未形成可执行任务|无正式 task 或仅初始输入态|Analysis 输入区、Inputs Tab、Scene Facts|发起分析、上传资产、绑定输入、向 AI 提问|Run Analysis|Workbench|
|Understanding|系统正在理解需求与识别分析类型|COGNIZING / ANALYSIS_SELECTION|AI 对话区、Analysis Summary、Current State|等待、补充说明、继续提问|运行、导出|Workbench|
|Planning|系统正在声明输入、组织执行方案|PLANNING|Input & Planning Summary、Task Stage、所需输入|继续提问、查看所需输入|正式运行（如未 ready）|Workbench|
|Validation Blocked|存在缺失输入或无效绑定，当前不能继续|VALIDATING / WAITING_USER|Missing Inputs、Invalid Bindings、Suggested Fix、Can Resume|上传、绑定、替换、Resume、Ask AI|Run Analysis、导出结果|Workbench + Governance|
|Ready to Run|输入和校验已满足，可进入执行|VALIDATED / READY|Analysis Type、Model、Required Inputs Ready、Run 主按钮|Run Analysis、查看摘要、Ask AI|修改核心对象时需触发重新校验|Workbench|
|Queued|已提交执行，正在排队|QUEUED|Current State、Lifecycle、预计进入运行、Cancel 可用性|查看任务详情、可选 Cancel、继续提问|修改关键输入、重新运行|Governance / Workbench Header|
|Running|正在执行分析|RUNNING / PRE_PROCESSING|Progress、Current Stage、Cancel、运行说明、结果占位|查看运行进度、可选 Cancel、继续提问|修改输入、重复提交、导出最终结果|Governance 为主，Workbench 辅助|
|Result Processing|运行已结束，系统正在整理结果与解释|RESULT_PROCESSING / ARTIFACT_PROMOTING|Result Bundle preparing、Explanation generating、Artifacts preparing|查看处理中状态、等待、继续提问|重复提交、最终导出（未完成前）|Governance / Report 占位|
|Succeeded|分析已完成，可查看和导出结果|SUCCEEDED|结果地图、指标、解释、导出入口、关联 Task/Scene|查看结果、导出、返回 Scene、继续追问解释|Resume（通常不需要）|Report Detail + Workbench|
|Failed Recoverable|分析失败，但可通过补充/修复继续|FAILED with can_resume=true|Failure Info、Suggested Fix、What’s Needed to Resume|Fix、Resume、Ask AI、Open Governance|导出最终结果|Governance|
|Failed Terminal|分析失败，当前轮次不可恢复|FAILED with can_resume=false|Failure Info、失败阶段、建议重新发起|查看日志摘要、复制配置重新发起、Ask AI|Resume|Governance|
|Cancelled|任务被取消|CANCELLED|Cancel 状态、取消发生阶段、可重新发起入口|Re-run、返回 Scene、Ask AI|Resume 当前轮次|Governance / Workbench|

|状态|展示重点|交互|
|---|---|---|

### 6.3.4 Analysis Tab 状态流转

|状态|对话区重点|不应出现|
|---|---|---|

|Succeeded|提供结果解释、支持追问|让解释替代结构化结果面板|
|Failed Recoverable|解释失败原因和恢复建议|用自由文本取代结构化治理对象|

|状态|展示要求|

|当前阶段|高亮 current|

### 6.5.1 Report List 状态流转

|状态|页面行为|
|---|---|
|尚无成功结果|列表为空，提示从 Scene/Task 发起分析|

|Failed to Prepare Report|显示结果可用性受限，建议转到 Task Governance 查看详情|

## 6.6 页面跳转与停留规则

|场景|推荐行为|
|---|---|
|用户刚发起分析|停留在 Workbench，不强制跳转|
|进入 WAITING_USER|Workbench 局部提示 + 可一键进入 Governance|
|已提交运行|Workbench 保持可见，但推荐用户可进入 Governance 查看全过程|
|正在运行|可停留 Workbench；如需详细阶段信息，引导到 Governance|
|结果处理中|Governance 与 Results 占位都可见，但不强制跳转|
|成功完成|提供显著入口进入 Report Detail，同时 Workbench 也可加载结果图层|
|恢复失败任务|从 Governance 或 Workbench 都可触发，但恢复后回到 Workbench 更自然|
