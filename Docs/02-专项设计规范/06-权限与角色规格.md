## 1. 文档目的
本文用于从前端产品与交互层面，定义不同用户角色在系统中的可见范围、可操作范围和页面进入范围。

本文重点回答以下问题：

1. 哪些页面和模块是普通用户默认可见的。
    
2. 哪些内容应只对高级用户开放。
    
3. 哪些治理与审计内容应只对运维 / 审计角色开放。
    
4. 同一页面在不同角色下应如何降级或展开。
    
5. 高级模式、审计模式应如何进入，如何避免干扰普通分析流程。
    

本文不定义完整的后端 RBAC 实现，也不定义真实权限字段，仅从前端视图层与交互层给出角色分层规则。

---

## 2. 适用范围与使用原则

### 2.1 适用范围

本文适用于以下页面与区域：

- Scenes / Scene Overview
    
- Scene Workbench
    
- Tasks
    
- Task Detail / Governance
    
- Manifest Tab
    
- Audit Tab
    
- Assets / Asset Detail
    
- Results / Result Detail
    
- Advanced View
    
- 审计相关区域
    
- 工件与结果容器相关区域
    

### 2.2 使用原则

权限与角色设计必须遵循以下原则：

#### 2.2.1 主流程优先

普通用户应能够在不接触内部治理细节的情况下，完成：

- 发起分析
    
- 上传和绑定输入
    
- 运行分析
    
- 查看状态
    
- 处理待补输入
    
- 查看结果与解释
    

#### 2.2.2 高级信息按需展开

更复杂的内容，例如：

- Manifest 扩展摘要
    
- Runtime Checks 摘要
    
- 更完整的事件流
    
- 工件高级元信息
    

应通过高级模式或角色差异展开，而不是默认堆给所有用户。

#### 2.2.3 审计内容与普通分析流分离

审计信息应满足：

- 结构化
    
- 可筛选
    
- 可回溯
    
- 不干扰普通分析主流程
    

普通用户不应默认进入 Audit 视图。

#### 2.2.4 角色差异不破坏主页面结构

角色变化应主要影响：

- 看得到什么
    
- 能点什么
    
- 能展开到多深
    

而不应让同一系统对不同角色变成完全不同的信息架构。

## 3. 角色分层

前端建议至少区分三类角色。

## 3.1 普通分析用户

### 定位

负责完成日常分析任务的主要用户。

### 典型职责

- 发起分析
    
- 补充输入
    
- 查看任务状态
    
- 处理 waiting / resume
    
- 查看结果与解释
    

### 设计原则

- 应以业务语义和任务流程为主
    
- 不要求理解内部编排和治理细节
    
- 不默认暴露审计和复杂运行信息
    

---

## 3.2 高级用户

### 定位

需要更完整执行上下文、参数摘要与结果解释能力的资深用户。

### 典型职责

- 深入查看分析上下文
    
- 理解 Manifest 摘要和运行检查摘要
    
- 查看更完整参数分组
    
- 处理较复杂的问题定位
    

### 设计原则

- 在普通用户基础上增加“更多摘要”
    
- 不等于直接暴露后台原始对象
    
- 高级模式应是显式开启，而不是默认铺开
    

---

## 3.3 运维 / 审计用户

### 定位

关注任务治理、事件流、失败排查、工件与审计信息的角色。

### 典型职责

- 查看完整生命周期与事件流
    
- 查看审计记录
    
- 查看失败或取消阶段的详细治理事实
    
- 查看更完整的工件与结果容器信息
    

### 设计原则

- 可见性更深，但仍保持结构化与摘要化
    
- 审计信息不应与普通分析界面混杂
    
- 重点服务于治理、排查与追溯

## 6. 页面级角色差异规则

## 6.1 Scene Workbench

### 普通分析用户

应看到：

- Header 摘要
    
- Inputs / Layers / Map
    
- Analysis Tab
    
- Task Tab 基础版
    
- Context Tab 摘要版
    

不应默认看到：

- 高级 Manifest 细项
    
- Runtime Checks 细项
    
- Workflow / Template 深层信息
    

### 高级用户

在普通用户基础上，可额外看到：

- 更完整的参数摘要
    
- Runtime Profile 摘要
    
- Validation Summary
    
- Context Tab 扩展信息
    
- Advanced View 入口
    

### 运维 / 审计用户

在高级用户基础上，可更容易进入：

- Governance 深层信息
    
- 更完整事件上下文
    
- 更多运行期摘要
    

但 Workbench 仍不应被做成运维后台界面。

---

## 6.2 Task Detail / Governance

### 普通分析用户

应看到：

- Top Summary
    
- Lifecycle
    
- Governance Panel
    
- Events 摘要
    
- Manifest 摘要
    
- Artifacts / Result Bundle 基础信息
    

不应默认看到：

- Audit Tab
    
- Manifest 原始对象
    
- 工件元信息细项
    

### 高级用户

可额外看到：

- 更完整的 Manifest 摘要
    
- 更完整 Events
    
- Runtime Checks 摘要
    
- Advanced View 下的治理辅助信息
    

### 运维 / 审计用户

应看到：

- Audit Tab
    
- 更完整事件流
    
- 更完整 Artifacts / Result Bundle 摘要
    
- 更充分的失败 / 取消 / 结果处理中治理信息
    

---

## 6.3 Result Detail

### 普通分析用户

应看到：

- Result Summary
    
- Map
    
- Metrics
    
- Explanation
    
- 导出入口
    
- 关联 Task / Scene 摘要
    

### 高级用户

可额外看到：

- 更完整的结果口径参数摘要
    
- Validation / Runtime 摘要的次级展开
    
- 更多输入快照说明
    

### 运维 / 审计用户

可额外看到：

- 与结果生成相关的更多工件摘要
    
- 更多运行结果容器信息
    

但不应让结果页变成审计页。

---

## 6.4 Assets / Asset Detail

### 普通分析用户

应看到：

- 资产列表
    
- Metadata 摘要
    
- Map Preview
    
- Used by Scenes / Tasks 摘要
    
- Binding History 摘要
    

### 高级用户

可额外看到：

- 更完整绑定历史
    
- 更完整元数据摘要
    

### 运维 / 审计用户

可额外看到：

- 更多治理相关关系信息
    
- 更完整使用链路
    

---

## 7. 同页分层规则

同一页面在不同角色下，应优先通过“展开深度”而非“重建页面结构”来区分。

## 7.1 普通模式

普通用户看到的页面应满足：

- 信息更聚焦于业务语义
    
- 下一步操作明确
    
- 技术复杂度尽量被折叠
    

## 7.2 高级模式

高级模式应增加：

- 更完整上下文
    
- 更完整参数摘要
    
- 更丰富 Manifest / Runtime 信息
    

但仍然保持摘要化和结构化。

## 7.3 审计 / 运维模式

审计 / 运维模式应增加：

- 事件流深度
    
- 审计记录
    
- 工件深度信息
    
- 失败与取消阶段的治理信息
    

但也不应直接把后台实现原文暴露给用户。

---

## 8. 高级模式入口规则

高级模式应作为一种**显式模式切换**存在，而不是默认把所有高级信息平铺给用户。

### 8.1 建议入口位置

可在以下页面提供 `Advanced View` 开关：

- Scene Workbench
    
- Task Detail / Governance
    
- Result Detail
    

### 8.2 开启后的行为

开启后可展开：

- 更完整的参数摘要
    
- Manifest 扩展信息
    
- Runtime Checks 摘要
    
- Validation Summary
    
- 更多上下文摘要
    

### 8.3 不应发生的情况

- 高级模式打开后页面结构完全重构
    
- 高级模式让页面变成后台对象原文展示
    
- 普通用户不理解自己为什么看不到某些内容且没有解释
    

---

## 9. 审计内容展示原则

审计内容需要满足：

- 结构化
    
- 可筛选
    
- 可回溯
    
- 不干扰普通用户主流程
    

## 9.1 审计内容适合展示什么

- who
    
- when
    
- did what
    
- on which object
    

## 9.2 审计内容不适合展示什么

- 大量无解释的内部技术细节
    
- 与当前用户无关的系统实现噪声
    
- 需要后台知识才能理解的原始记录全文
    

## 9.3 审计内容入口规则

- 普通用户不默认进入 Audit Tab
    
- 高级用户可在受控条件下查看
    
- 运维 / 审计用户默认可见
    

---

## 10. 信息隐藏原则

角色化设计的关键，不只是“谁能看什么”，还包括“哪些内容无论如何都不该作为主概念呈现”。

以下内容默认不应作为用户主界面概念：

- Skill
    
- Selected Skill
    
- capability key
    
- graph digest
    
- internal prompt version
    
- 思维流 / 推理链
    

即使是运维 / 审计角色，也应优先看到经过整理的结构化摘要，而不是直接看到原始内部对象。

---

## 11. 角色与状态的组合原则

角色差异不应破坏状态主语义。

无论是什么角色，以下状态语义必须保持一致：

- Waiting for Required Input
    
- Ready to Run
    
- Running
    
- Processing Results
    
- Completed
    
- Failed
    
- Cancelled
    

也就是说：

- 高级用户可以看得更深
    
- 运维用户可以看得更全
    

但他们看到的主状态不应和普通用户不同。

---

## 12. 角色相关的验收要点

## 12.1 普通分析用户验收

- 可完成完整分析主流程
    
- 不默认看到 Audit 与内部编排信息
    
- 可查看结果与解释
    
- 遇到 waiting / failed 时有明确处理路径
    

## 12.2 高级用户验收

- 可开启 Advanced View
    
- 可看到更完整 Manifest / Validation / Runtime 摘要
    
- 页面仍保持可读，不变成后台对象浏览器
    

## 12.3 运维 / 审计用户验收

- 可查看 Audit Tab
    
- 可查看更完整 Events / Artifacts / Result Bundle 摘要
    
- 可追踪失败、取消、结果处理中阶段的治理信息
    

---

## 13. 推荐的最小实现顺序

如果准备分阶段实现角色化前端，推荐顺序如下：

1. 先实现普通用户视角的完整主流程。
    
2. 再补高级模式入口与高级摘要展开。
    
3. 然后补 Audit Tab 与更完整事件流。
    
4. 最后再细化下载权限、取消权限、工件深度可见性。
    

---

## 14. 使用建议

### 给设计师

- 先画普通模式，再设计高级模式展开。
    
- 不要一开始就把所有角色信息叠在同一层。
    
- 同页差异优先通过“信息展开深度”表达。
    

### 给前端

- 角色差异优先体现在 View 层和交互入口层。
    
- 不要让页面逻辑因为角色差异变成多套完全不同实现。
    
- 先保证主状态和主流程一致，再细化高级可见性。
    

### 给产品 / 架构

- 用本文约束“谁该看什么、谁不该看什么”。
    
- 不要把所有内部对象默认丢给前端。
    
- 用本文作为高级模式和审计模式讨论基准。
