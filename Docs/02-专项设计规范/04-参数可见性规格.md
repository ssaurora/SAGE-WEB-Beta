## 1. 文档目的

本文用于从前端产品与交互层面，定义系统中参数的展示边界、编辑边界和用户可见性分层规则。

本文重点回答以下问题：

1. 哪些参数用户应该看见。
    
2. 哪些参数只需要系统内部持有。
    
3. 哪些参数可以编辑，哪些只能查看。
    
4. 哪些参数适合在普通模式显示，哪些只应在高级模式显示。
    
5. 参数区与 AI 对话区各自负责什么，不能互相替代什么。
    

本文不定义真实后端字段名，也不假设具体接口实现，仅定义参数在前端中的产品层分类、展示规则与交互约束。

---

## 2. 适用范围与使用原则

### 2.1 适用范围

本文适用于以下页面与区域：

- Scene Workbench
    
- Workbench Header
    
- Inputs Tab
    
- Analysis Tab
    
- Context Tab
    
- Task Detail / Governance
    
- Manifest Tab
    
- Result Detail
    
- Analysis Parameters Panel（如后续实现）
    
- 高级模式 / 审计模式下的参数摘要区
    

### 2.2 使用原则

前端参数设计必须遵循以下原则：

#### 2.2.1 用户理解优先

参数展示的首要目标，是让用户理解：

- 当前分析到底基于什么条件运行
    
- 当前结果对应什么输入和约束
    
- 修改哪些内容会影响可运行性和结果口径
    

#### 2.2.2 不把界面做成原始 schema 表单

前端不应把底层参数 schema 原样铺开给用户。  
参数必须经过产品语义整理后再展示。

#### 2.2.3 参数必须分层展示

不同类型的参数应采用不同展示策略，而不是一律“展示全部”或“一律隐藏”。

#### 2.2.4 参数区与 AI 对话区职责分离

参数区负责展示和正式修改。  
AI 对话区负责解释、建议和答疑。  
AI 不能替代参数区成为参数的唯一权威来源。

---

## 3. 参数分类模型

前端应将参数拆成四类，而不是把所有参数平铺为同一层级。

## 3.1 A 类：用户分析参数（User Analysis Parameters）

这类参数直接影响分析目标、范围、口径或用户预期。

典型示例：

- Analysis Type
    
- Study Area / Region
    
- Scenario / Case Selection
    
- Time Scope（如适用）
    
- 关键分析约束
    
- 用户显式指定的输出偏好
    

### 设计原则

- 这类参数应优先可见。
    
- 这类参数应优先使用业务语义命名。
    
- 这类参数通常是普通用户最关心的参数。
    

---

## 3.2 B 类：模型业务参数（Model-facing Business Parameters）

这类参数与模型业务含义有关，但未必需要普通用户全量调节。

典型示例：

- 阈值类参数
    
- 输出聚合层级
    
- 业务计算模式
    
- 某类分析的业务规则设置
    
- 影响结果口径的关键业务项
    

### 设计原则

- 这类参数应选择性可见。
    
- 普通用户只需看到关键摘要。
    
- 高级模式可展开更完整内容。
    

---

## 3.3 C 类：运行参数（Execution Parameters）

这类参数与运行时相关，但不是普通用户的主关注点。

典型示例：

- Execution Mode
    
- Runtime Profile
    
- Worker Profile
    
- 资源等级选择
    
- 输出策略摘要
    

### 设计原则

- 普通模式通常只显示摘要或只读信息。
    
- 高级模式可展示更完整运行上下文。
    
- 普通用户不应被迫理解这些参数。
    

---

## 3.4 D 类：内部编排参数（Internal Orchestration Parameters）

这类参数服务于编排、校验、修复、规划或调度，不适合普通用户直接看见。

典型示例：

- Workflow / Template 内部标识
    
- planning revision 细项
    
- graph digest
    
- capability key
    
- internal prompt version
    
- runtime assertion internal details
    

### 设计原则

- 默认不在普通界面中作为参数展示。
    
- 高级模式也应只显示摘要，不应直接暴露内部实现原文。
    
- 不应成为普通用户主界面的一等概念。
    

---

## 4. 参数可见性分层

参数展示应至少区分三层模式。

## 4.1 普通模式（Default Mode）

普通模式下，页面应重点展示：

- 用户分析参数（A 类）
    
- 少量关键业务参数（B 类）
    
- 输入条件摘要
    
- 当前状态与执行上下文摘要
    

### 普通模式目标

- 让普通用户知道“当前分析是什么、用什么数据、处于什么状态、结果基于什么条件”
    
- 避免信息过载
    
- 保持分析流程清晰
    

---

## 4.2 高级模式（Advanced Mode）

高级模式下，可在普通模式基础上增加展示：

- 更完整的业务参数（B 类）
    
- 执行上下文参数（C 类）
    
- Manifest 摘要扩展项
    
- Runtime Checks 摘要
    
- Validation Summary
    
- Planning Revision 摘要
    

### 高级模式目标

- 服务资深分析用户与项目负责人
    
- 帮助理解分析为何这样运行
    
- 但仍然保持摘要化、可解释，而不是直接铺后台对象
    

---

## 4.3 调试 / 审计模式（Debug / Audit Mode）

调试或审计模式下，才允许受控查看：

- 更细的运行检查项
    
- 工件元信息
    
- 更完整的执行上下文
    
- 编排与版本摘要
    

### 调试 / 审计模式原则

- 仍不直接展示思维流
    
- 不将内部 prompt 细节作为主界面信息
    
- 仅面向高权限角色或特定场景
    

---

## 5. 参数编辑权限分层

参数编辑不应只有“能改/不能改”两种，而应至少区分为四种状态。

|编辑状态|含义|适用场景|
|---|---|---|
|Editable|用户可直接修改|用户分析参数、少量关键业务参数|
|Editable with Revalidation|可修改，但修改后必须重新校验|关键输入相关参数、业务阈值、输出口径|
|Read-only Visible|可查看但不可编辑|Runtime Profile、Execution Context、Manifest Summary|
|Hidden|默认不展示|graph digest、capability key、内部 revision 细项|

### 设计原则

- 是否可编辑必须有明确 UI 表达。
    
- 用户修改后是否需要重新校验必须明确提示。
    
- 隐藏参数不应被 AI 对话区间接“偷偷暴露”为主概念。
    

---

## 6. 各页面的参数可见性规则

## 6.1 Scene Workbench

### 6.1.1 Workbench Header

应展示：

- Analysis Type
    
- Model
    
- Current State
    
- Required Inputs Ready
    

可选展示：

- 关键分析范围标签
    
- 当前场景标签
    

不应展示：

- 内部编排参数
    
- 长参数列表
    
- 底层 schema 片段
    

### 设计目标

Header 应承担“当前分析摘要”角色，而不是参数详情区。

---

### 6.1.2 Inputs Tab

应展示：

- 输入角色及绑定状态
    
- 输入类型要求
    
- 缺失/无效原因摘要
    

不应展示：

- 底层 schema 原文
    
- 内部验证规则细节
    
- 运行参数
    

### 设计目标

Inputs Tab 应服务于“我需要什么、我绑定了什么、哪里有问题”，而不是承担参数面板职责。

---

### 6.1.3 Analysis Tab

应展示：

- 用户当前意图摘要
    
- 系统识别出的 Analysis Type
    
- Model
    
- Missing Inputs
    
- Suggested Next Steps
    
- 少量关键可调分析约束
    

高级模式可增加：

- 业务参数摘要
    
- Workflow / Template（次级）
    
- 执行模式摘要
    

不应展示：

- Selected Skill
    
- capability key
    
- graph digest
    

### 设计目标

Analysis Tab 负责“解释分析是什么”和“引导用户下一步”，而不是变成复杂参数设置区。

---

### 6.1.4 Context Tab

应展示：

- Scene Facts
    
- Bound Inputs
    
- Execution Context（摘要）
    
- Manifest Summary（摘要）
    

高级模式可增加：

- Runtime Profile
    
- Validation Summary
    
- Runtime Checks 摘要
    

不应展示：

- 内部调度参数细项
    
- 长参数表单
    

### 设计目标

Context Tab 应承担“当前分析上下文快照”角色。

---

## 6.2 Task Detail / Governance

### 6.2.1 Top Summary

应展示：

- Analysis Type
    
- Model
    
- Current State
    
- Planning Revision（摘要）
    
- Checkpoint Version（摘要）
    

不应展示：

- 全部参数组
    
- 原始执行对象结构
    

---

### 6.2.2 Governance Panel

应展示：

- 缺失输入
    
- 无效绑定
    
- 可恢复性
    
- 需要的下一步动作
    

不应把“参数编辑”直接塞进 Governance Panel。  
需要改参数时，应回到 Workbench 或进入独立参数面板。

### 设计目标

Governance Panel 是治理区，不是参数配置区。

---

### 6.2.3 Manifest Tab

普通模式展示：

- Manifest Status
    
- Bound Inputs
    
- Validation Status
    
- Runtime Checks（摘要）
    

高级模式可增加：

- 更完整的 Manifest 摘要项
    
- 执行上下文参数摘要
    

不应直接展示：

- 完整内部对象 JSON
    
- 编排细节原文
    

---

## 6.3 Result Detail

应展示：

- 结果解释依赖的关键分析参数摘要
    
- 输出口径相关参数摘要
    
- 输入快照摘要
    

不应展示：

- 全部运行参数
    
- 内部图编排参数
    
- 大量不影响结果理解的技术参数
    

### 设计目标

结果页应回答“这个结果在什么条件下得到”，而不是让用户面对执行对象全文。

---

## 7. 参数面板设计建议

若后续需要真正提供可调参数入口，建议不要把参数散落在多个区域，而应引入独立的 **Analysis Parameters Panel**（抽屉或弹窗）。

## 7.1 参数面板目标

用于集中承载：

- 用户可编辑的分析参数
    
- 关键业务参数
    
- 修改后的重新校验提示
    

## 7.2 参数面板不应承载

- 输入绑定本身
    
- 运行日志
    
- 内部编排对象
    
- 审计信息
    

## 7.3 参数面板分区建议

|分区|内容|
|---|---|
|Analysis Scope|研究范围、分析主题、场景选择|
|Business Parameters|关键业务参数、输出口径、阈值|
|Execution Preferences|少量用户可感知的运行偏好|
|Revalidation Notice|修改后会触发哪些重新校验|

---

## 8. 参数在不同状态下的编辑规则

参数编辑权限必须随任务状态变化。

|主状态|参数编辑建议|
|---|---|
|Draft / Understanding|可编辑|
|Planning|可编辑，但部分变更会触发重新规划|
|Waiting for Required Input|可编辑，且应强调修改后重新校验|
|Ready to Run|可编辑，但关键修改会使状态退回待校验|
|Queued|本轮参数只读|
|Running|本轮参数只读|
|Processing Results|本轮参数只读|
|Completed|本轮参数只读；允许复制为新分析|
|Failed Recoverable|与失败原因相关的参数可重新开放编辑|
|Failed Terminal|本轮只读；建议复制后重新发起|
|Cancelled|本轮只读；允许复用为新分析|

### 设计原则

- 不允许运行中修改本轮关键参数。
    
- Completed 后查看的是“结果对应的参数快照”，不是仍在生效的可编辑配置。
    
- Failed Recoverable 必须清楚区分“可改哪些”和“为什么改”。
    

---

## 9. 参数修改后的用户反馈规则

当用户修改关键参数时，前端必须明确反馈其影响。

## 9.1 应明确提示的影响类型

- 需要重新校验输入
    
- 会影响当前可运行性
    
- 会改变结果口径
    
- 会使当前已完成运行结果不再对应最新配置
    

## 9.2 推荐提示文案示例

- `Changing this setting will require validation again.`
    
- `This change affects the analysis configuration and may require re-running the task.`
    
- `The current completed result reflects the previous configuration.`
    

## 9.3 不应发生的情况

- 用户改了关键参数，但界面没有任何状态变化
    
- 旧结果仍被伪装成和新参数一致
    
- AI 说需要重校验，但结构化 UI 没有同步变化
    

---

## 10. 用户可见的最小参数集合

对于普通模式，建议每类分析至少保证用户能看到以下最小参数集合：

|参数类别|最小可见内容|
|---|---|
|分析识别|Analysis Type、Model|
|分析范围|Scene / Region / Study Area|
|输入条件|Required Inputs、Bound Inputs、Missing Inputs|
|业务设置|少量关键业务参数摘要|
|运行上下文|Current State、Execution Context 摘要|
|结果口径|影响结果解释的关键参数摘要|

### 设计原则

如果某类分析做不到这套最小集合，用户会感觉系统是黑箱。

---

## 11. 高级模式最小扩展集合

高级模式建议额外展示：

- Workflow / Template（次级）
    
- Runtime Profile
    
- Validation Summary
    
- Runtime Checks 摘要
    
- Planning Revision 摘要
    
- 更完整的业务参数组
    

### 设计原则

这些对象必须保持“摘要化、可解释”，而不是直接变成后台对象原文展示。

---

## 12. 参数与 AI 对话的职责边界

参数区和 AI 对话区不能互相替代。

## 12.1 参数区负责

- 展示当前有效参数
    
- 告知是否可编辑
    
- 告知修改后果
    
- 提供正式修改入口
    

## 12.2 AI 对话区负责

- 解释参数是什么意思
    
- 回答“为什么建议这样设置”
    
- 辅助用户决定是否修改
    
- 解释某个参数变化对结果的影响
    

## 12.3 不应由 AI 单独负责的内容

- 当前有效参数的唯一权威展示
    
- 可编辑性状态
    
- 修改后是否已生效
    

这些必须由结构化 UI 给出。

---

## 13. 推荐的最小实现顺序

如果你准备分阶段实现参数可见性，推荐顺序如下：

1. 先实现最小参数集合：Analysis Type、Model、Required Inputs、Bound Inputs、Current State、结果口径摘要。
    
2. 再实现参数面板：提供少量关键业务参数编辑入口。
    
3. 然后实现高级模式：Runtime Profile、Validation Summary、Planning Revision 摘要。
    
4. 最后再考虑更细的参数分组与用户角色差异。
    

---

## 14. 使用建议

### 给设计师

- 先按参数分类模型做信息分层。
    
- 不要把参数页画成底层 schema 表单。
    
- 普通模式与高级模式应有明显差异。
    

### 给前端

- 先实现参数展示，再实现参数编辑。
    
- 不要把参数有效性判断全部放在组件内部推理。
    
- 修改参数后的状态回退与重新校验提示必须与主状态联动。
    

### 给产品 / 架构

- 用本文约束“哪些参数属于用户应该知道的，哪些属于系统内部”
    
- 用本文作为参数展示边界与高级模式讨论的基准
