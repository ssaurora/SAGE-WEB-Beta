## 1. 文档目的

本文用于从前端产品与交互层面，定义系统在不同任务状态和运行状态下的页面表现规则。

本文重点回答以下问题：

1. 不同状态下，页面应重点展示什么。
    
2. 不同状态下，用户能做什么、不能做什么。
    
3. 状态切换时，哪些页面区域必须同步联动。
    
4. Workbench、Governance、Report 等页面在不同阶段分别承担什么职责。
    
5. 运行中状态如何表达，避免用户误解任务已完成或已失败。
    

本文不定义真实后端字段名，也不假设具体接口实现，仅定义前端页面层的状态语义、展示方式与交互约束。

---

## 2. 适用范围与使用原则

### 2.1 适用范围

本文适用于以下页面与区域：

- Scene Workbench
    
- Task Detail / Governance
    
- Reports / Report Detail
    
- Workbench Header
    
- Inputs Tab
    
- Analysis Tab
    
- Task Tab
    
- Context Tab
    
- Lifecycle Panel
    
- Result 占位区域
    

### 2.2 使用原则

前端状态设计必须遵循以下原则：

#### 2.2.1 主状态优先

页面首先应向用户表达**当前任务主状态**，例如：

- Waiting for Required Input
    
- Ready to Run
    
- Running
    
- Processing Results
    
- Completed
    
- Failed
    

而不是优先暴露底层细粒度状态。

#### 2.2.2 阶段说明次之

在主状态之下，可补充阶段说明，用于增强用户理解，例如：

- Preparing workspace
    
- Running preprocessing
    
- Running model
    
- Packing results
    
- Generating explanation
    

#### 2.2.3 页面联动一致

同一任务状态变化时，Header、Task Tab、Lifecycle、Report 占位等区域必须同步更新，避免一处显示“运行中”、另一处显示“已完成”。

#### 2.2.4 结构化事实优先于 AI 文本解释

AI 对话区可以解释状态，但不能替代权威状态展示。以下信息必须由结构化 UI 明确表达：

- Current State
    
- Can Resume
    
- Can Cancel
    
- Result Available
    
- Missing Required Input
    
- Invalid Bindings
    

---

## 3. 状态展示分层

前端状态应分为三层：

### 3.1 主状态（Primary State）

用户当前最需要知道的任务状态，例如：

- Draft
    
- Understanding
    
- Planning
    
- Waiting for Required Input
    
- Ready to Run
    
- Queued
    
- Running
    
- Processing Results
    
- Completed
    
- Action Required
    
- Failed
    
- Cancelled
    

### 3.2 阶段状态（Stage State）

用于说明当前运行或流程推进到了哪一步，例如：

- COGNIZING
    
- PLANNING
    
- VALIDATING
    
- QUEUED
    
- RUNNING
    
- RESULT_PROCESSING
    
- ARTIFACT_PROMOTING
    

以及运行阶段提示，例如：

- Preparing workspace
    
- Materializing inputs
    
- Running preprocessing
    
- Running model
    
- Running checks
    
- Packing results
    

### 3.3 局部状态（Local UI State）

指页面某个局部组件的状态，例如：

- Inputs Tab 缺失输入
    
- Map 暂无图层
    
- Analysis Tab 正在等待 AI 回复
    
- Report Detail 正在 preparing
    

局部状态不能与主状态冲突。

---

## 4. 用户侧主状态定义

建议前端统一使用以下主状态语义：

|主状态|用户语义|
|---|---|
|Draft / Not Started|尚未形成正式分析任务|
|Understanding|系统正在理解需求与识别分析类型|
|Planning|系统正在组织分析方案与输入声明|
|Waiting for Required Input|当前缺少必需输入或存在无效绑定|
|Ready to Run|已满足运行前条件|
|Queued|任务已提交，等待执行|
|Running|任务正在执行|
|Processing Results|执行已结束，结果仍在整理|
|Completed|结果已可消费|
|Action Required|任务未完成，需要用户处理后继续|
|Failed|当前任务失败|
|Cancelled|当前任务已取消|

说明：

- `Action Required` 更适合用于可恢复失败或待补输入场景。
    
- `Processing Results` 必须作为独立主状态存在，不能混同为 Running 或 Completed。
    
- `Cancelled` 必须与 Failed 区分。

|---|---|---|---|---|---|
|Draft / Not Ready|无正式 task 或仅初始输入态|Analysis 输入区、Inputs Tab、Scene Facts|发起分析、上传资产、绑定输入、向 AI 提问|Run Analysis|Workbench|
|Understanding|COGNIZING / ANALYSIS_SELECTION|Analysis Summary、Current State|等待、补充说明、继续提问|运行、导出|Workbench|
|Planning|PLANNING|Input & Planning Summary、所需输入|继续提问、查看所需输入|正式运行（如未 ready）|Workbench|
|Waiting for Required Input|VALIDATING / WAITING_USER|Missing Inputs、Invalid Bindings、Suggested Fix、Can Resume|上传、绑定、替换、Resume、Ask AI|Run Analysis、导出结果|Workbench + Governance|
|Ready to Run|VALIDATED / READY|Analysis Type、Model、Required Inputs Ready、Run 主按钮|Run Analysis、查看摘要、Ask AI|修改核心对象后仍保留 ready 状态|Workbench|
|Queued|QUEUED|Current State、Lifecycle、排队说明|查看任务详情、可选 Cancel、继续提问|修改关键输入、重新运行|Governance / Header|
|Running|RUNNING / PRE_PROCESSING|Current Stage、Progress、Cancel、运行说明|查看进度、可选 Cancel、继续提问|修改输入、重复提交、导出结果|Governance 为主，Workbench 辅助|
|Processing Results|RESULT_PROCESSING / ARTIFACT_PROMOTING|Result Bundle preparing、Explanation generating、Artifacts preparing|查看处理中状态、等待、继续提问|提前导出、重复提交|Governance / Report 占位|
|Completed|SUCCEEDED|结果地图、指标、解释、导出入口|查看结果、导出、返回 Scene、继续追问解释|Resume|Report Detail + Workbench|
|Failed Recoverable|FAILED + can_resume=true|Failure Info、Suggested Fix、What’s Needed to Resume|Fix、Resume、Ask AI、Open Governance|导出正式结果|Governance|
|Failed Terminal|FAILED + can_resume=false|Failure Info、失败阶段、重新发起建议|查看失败摘要、复制配置重发、Ask AI|Resume|Governance|
|Cancelled|CANCELLED|取消状态、取消发生阶段、重新发起入口|Re-run、返回 Scene、Ask AI|Resume 当前轮次|Governance / Workbench|

## 6.1 Scene Workbench 页面状态流转

### 6.1.1 Workbench 总体流转

|Workbench 状态|触发条件|Header 表现|Inputs Tab|Analysis Tab|Task Tab|Context Tab|主按钮|

|Input Required|已识别分析，但缺输入|Current State = Waiting for Required Input|强调 Missing Required Inputs|展示 Missing Inputs 与建议|显示 Can Resume、Suggested Fix|Manifest 区弱化或显示未就绪|Complete Required Inputs|

|Queued|已提交执行，等待运行|Current State = Queued|锁定关键绑定编辑|展示等待说明|显示阶段链路到 QUEUED|显示 Execution Context|View Task / Cancel|

|Processing Results|结果整理中|Current State = Processing Results|只读|展示“结果正在准备”|显示 RESULT_PROCESSING|显示 Result Bundle preparing|View Task|
|Completed|分析完成|Current State = Completed|只读或允许复制为新分析|展示结果解释入口|显示已完成|显示最终 Manifest / Execution Context|View Results|
|Failed Recoverable|失败但可恢复|Current State = Action Required|高亮问题输入/绑定|展示失败解释与修复建议|强调 Can Resume / Suggested Fix|显示失败前上下文快照|Fix and Resume|

|Cancelled|本轮已取消|Current State = Cancelled|显示本轮输入快照|展示取消说明与下一步建议|显示取消发生阶段|上下文只读|Start New Analysis / Re-run|

### 6.1.3 Inputs Tab 状态流转

|状态|展示重点|交互|
|---|---|---|
|Idle|已有资产与推荐输入角色|可上传、可预绑定|
|Understanding|若已识别 required roles，则显示角色占位|可继续上传/绑定|
|Input Required|Missing Required Inputs 强高亮；Invalid Bindings 强高亮|Bind / Upload / Replace 全开启|
|Ready|全部 required roles 已 Bound；可显示校验通过状态|允许查看；修改需触发重新校验|

|Completed|显示用于本轮结果的输入快照|可复制为新分析输入基础|
|Failed Recoverable|高亮失败相关输入/绑定|开启 Fix / Replace / Resume|
|Failed Terminal|显示失败轮次输入快照|允许复制配置重建|
|Cancelled|显示取消轮次输入快照|允许复制为新分析配置|

### 6.1.4 Analysis Tab 状态流转

|状态|对话区重点|不应出现|
|---|---|---|
|Idle|引导用户描述任务|无关闲聊欢迎流|
|Understanding|展示“系统理解到的分析任务”|技术性内部术语堆叠|
|Input Required|解释为什么缺输入、需要补什么|用对话替代上传/绑定操作|
|Ready|解释当前为何可运行、运行前注意事项|过度技术细节|
|Queued / Running|解释当前阶段、回答用户追问|假装已经有最终结果|
|Processing Results|解释“为什么还不能下载/查看最终结果”|提前承诺最终结果内容|
|Completed|提供结果解释、支持追问|让解释替代结构化结果面板|
|Failed Recoverable|解释失败原因和恢复建议|用自由文本取代结构化治理对象|
|Failed Terminal|给出失败总结和重新发起建议|虚假可恢复承诺|
|Cancelled|解释取消发生阶段与下一步建议|把取消伪装成失败或完成|

## 6.3 Reports / Result 页面状态流转

### 6.3.1 Report List 状态流转

|状态|页面行为|
|---|---|
|尚无成功结果|列表为空，提示从 Scene/Task 发起分析|
|Processing Results|可显示“结果准备中”占位卡片|
|Completed|正常展示报告列表|
|Failed / Cancelled|不生成正式报告卡片，必要时仅在 Task 中查看失败或取消摘要|

---

### 6.3.2 Report Detail 状态流转

|状态|页面行为|
|---|---|
|Processing|结果页以 preparing 状态占位，显示“结构化结果与解释正在准备中”|
|Completed|展示 Result Summary、Map、Metrics、Explanation、Exports|
|Partial Available（如未来支持）|先展示已准备好的指标/地图占位，并明确部分完成|
|Failed to Prepare Report|显示结果可用性受限，建议转到 Task Governance 查看详情|

## 6.4 页面跳转与停留规则

|场景|推荐行为|
|---|---|
|用户刚发起分析|停留在 Workbench，不强制跳转|
|进入 WAITING_USER|Workbench 局部提示 + 可一键进入 Governance|
|已提交运行|Workbench 保持可见，但推荐用户进入 Governance 查看全过程|
|正在运行|可停留 Workbench；如需详细阶段信息，引导到 Governance|
|结果处理中|Governance 与 Results 占位都可见，但不强制跳转|
|成功完成|提供显著入口进入 Report Detail，同时 Workbench 可加载结果图层|
|恢复失败任务|从 Governance 或 Workbench 都可触发，但恢复后回到 Workbench 更自然|
|取消任务后|停留 Governance 或回到 Workbench 均可，但应提供重新发起入口|

## 7. 运行中状态与进度规格

## 7.1 设计目标

运行中状态与进度设计需要满足四个目标：

1. 让用户知道任务确实在推进，而不是“卡住了”。
    
2. 让用户知道当前处于哪一阶段，而不是只看到一个笼统的 Running。
    
3. 让用户知道此时还能做什么，例如查看详情、取消、继续提问。
    
4. 让用户知道什么时候结果还未准备好，避免把“执行成功”和“结果已可消费”混为一谈。
    

---

## 7.2 运行状态分层

### 7.2.1 用户可见主状态

用户侧统一展示以下主状态：

- Queued
    
- Running
    
- Processing Results
    
- Completed
    
- Action Required
    
- Failed
    
- Cancelled
    

### 7.2.2 内部阶段提示

在主状态之下，可以补充展示更细的阶段提示，例如：

- Waiting for execution slot
    
- Preparing workspace
    
- Materializing inputs
    
- Running preprocessing
    
- Running model
    
- Running checks
    
- Packing results
    
- Preparing artifacts
    
- Generating explanation
    

这些内容应作为**阶段说明**出现，而不是替代主状态本身。

---

## 7.3 Run Status Card 规格

运行期间，Workbench 和 Governance 页面都应存在一个统一的 Run Status Card 或等价区域。

### 7.3.1 必须展示

- Primary State
    
- Current Stage
    
- Task ID
    
- Analysis Type
    
- Model
    
- 是否可取消
    
- 最近一次状态更新时间
    
- 当前可执行操作
    

### 7.3.2 可选展示

- 简要进度文案
    
- 结果准备情况
    
- 失败/取消提示
    
- 进入 Governance 的快捷入口
    

### 7.3.3 不应展示

- 原始日志滚动流作为主界面内容
    
- 大量内部运行指标直接堆叠
    
- 对最终结果的提前承诺
    

---

## 7.4 Queued 状态规格

### 7.4.1 用户语义

系统已经接受了任务，但尚未开始真正运行。

### 7.4.2 页面文案建议

- `Analysis task submitted`
    
- `Waiting to start execution`
    
- `Queued for execution`
    

### 7.4.3 页面表现

|页面|表现|
|---|---|
|Workbench Header|Current State = Queued；主按钮切换为 View Task；可显示 Cancel（如允许）|
|Inputs Tab|切为当前轮次输入快照，只读|
|Analysis Tab|说明任务已提交，当前在等待执行|
|Task Tab|高亮 QUEUED；说明已受理、尚未开始|
|Governance 页|Lifecycle 高亮 QUEUED；Events 可持续追加|
|Results 页|不可进入正式结果消费；如已建卡片，仅显示 preparing/queued 占位|

### 7.4.4 用户允许操作

- 查看任务详情
    
- 继续向 AI 询问当前状态
    
- 在允许时取消任务
    

### 7.4.5 用户限制操作

- 不允许修改本轮关键输入绑定
    
- 不允许重复提交同一轮次运行
    
- 不允许导出结果
    

---

## 7.5 Running 状态规格

### 7.5.1 用户语义

任务已经真正开始执行，但最终结果尚不可消费。

### 7.5.2 页面文案建议

- `Analysis is running`
    
- `Model execution in progress`
    
- `Current stage: Running preprocessing`
    
- `Current stage: Running model`
    

### 7.5.3 运行阶段子状态

|阶段提示|用户理解|
|---|---|
|Preparing workspace|正在准备本轮运行环境|
|Materializing inputs|正在整理或确认输入|
|Running preprocessing|正在执行预处理|
|Running model|正在执行正式分析|
|Running checks|正在执行运行时检查|
|Packing results|正在整理运行输出|

这些阶段提示应显示在：

- Workbench Header 的次级说明区
    
- Governance 页的 Run Status Card
    
- Lifecycle 或 Stage 区域
    

### 7.5.4 页面表现

|页面|表现|
|---|---|
|Workbench Header|Current State = Running；显示 Current Stage；主按钮为 View Task|
|Inputs Tab|本轮输入快照只读|
|Analysis Tab|用户可继续提问“当前在做什么”“结果什么时候能看”|
|Task Tab|显示 Current Workflow Stage、Task ID、运行中事实|
|Context Tab|显示冻结后的上下文摘要|
|Governance 页|作为运行期主页面，展示 Lifecycle、Events、Run Status Card|
|Results 页|若进入，仅显示运行中占位，不展示最终指标与导出|

### 7.5.5 用户允许操作

- 查看任务细节
    
- 继续提问
    
- 在允许时取消任务
    
- 离开页面后稍后再回看
    

### 7.5.6 用户限制操作

- 不允许修改本轮输入
    
- 不允许重新运行同一轮次
    
- 不允许把“运行中”当成“已完成”导出
    

---

## 7.6 进度表达规格

### 7.6.1 进度表达原则

本系统更适合采用**阶段进度**，而不是伪精确百分比。

原因：

- GIS 与生态分析任务时长不稳定
    
- 不同案例的 I/O 和计算量差异很大
    
- 百分比进度会制造虚假确定性
    

因此推荐：

- 主表达：当前阶段
    
- 辅表达：阶段链路完成情况
    
- 可选表达：最近状态更新时间
    

### 7.6.2 推荐形式

#### 形式 A：阶段标签

例如：

- Preparing workspace
    
- Running preprocessing
    
- Running model
    
- Packing results
    

#### 形式 B：阶段 Stepper

例如：

- Submitted
    
- Queued
    
- Running
    
- Processing Results
    
- Ready
    

#### 形式 C：阶段说明文本

例如：

- `The analysis has started and is currently running model execution.`
    
- `The run has finished. The system is now preparing outputs and explanations.`
    

### 7.6.3 不推荐形式

- 精确到个位数的百分比进度条，除非后端真的能提供可信进度
    
- 长日志窗口作为默认进度视图
    
- 只显示 spinning loader 而没有阶段说明
    

---

## 7.7 Heartbeat Lost / 状态不确定规格

如果系统后续支持 `HEARTBEAT_LOST` 或等价状态，前端应区分“运行中”与“状态不确定”。

### 7.7.1 用户语义

任务不一定失败，但系统暂时无法确认它是否还在正常推进。

### 7.7.2 页面文案建议

- `Run status is temporarily unavailable`
    
- `The system has not received a recent progress update`
    
- `Please check the task details`
    

### 7.7.3 页面表现

|页面|表现|
|---|---|
|Workbench Header|不再显示普通 Running，而是 warning 式状态|
|Task Tab|显示最近可确认状态时间|
|Governance 页|高亮 warning，提示状态不可判定|
|Results 页|不展示结果可用性承诺|

### 7.7.4 用户允许操作

- 查看 Governance 详情
    
- 向 AI 询问当前情况
    
- 依据策略可选取消或等待
    

### 7.7.5 用户限制操作

- 不应把此状态渲染成成功或失败
    
- 不应提前开放结果下载
    

---

## 7.8 Cancel 相关规格

## 7.8.1 Cancelling 状态

### 用户语义

系统已收到取消请求，但当前轮次尚未完全停止。

### 页面文案建议

- `Cancelling analysis`
    
- `The system is stopping the current run`
    

### 页面表现

|页面|表现|
|---|---|
|Workbench Header|Current State = Cancelling；按钮禁用或只保留 View Task|
|Analysis Tab|可解释“当前正在安全停止任务”|
|Governance 页|Lifecycle 标示为 cancelling 中间态|

### 用户限制

- 不允许再次点击取消
    
- 不允许重新运行同一轮次
    

## 7.8.2 Cancelled 状态

### 用户语义

任务已经停止，本轮不会继续产生产出。

### 页面文案建议

- `Analysis cancelled`
    
- `The current run was stopped before completion`
    

### 页面表现

|页面|表现|
|---|---|
|Workbench Header|Current State = Cancelled；主按钮改为 Start New Analysis 或 Re-run|
|Inputs Tab|显示本轮输入快照|
|Analysis Tab|AI 可解释取消发生在何阶段以及下一步怎么做|
|Governance 页|Lifecycle 标记取消位置；Events 完整保留|
|Results 页|通常不生成正式结果；若有局部中间结果，也不应伪装成正式报告|

### 用户允许操作

- 重新发起分析
    
- 返回 Scene
    
- 查看任务历史
    

---

## 7.9 Processing Results 状态规格

### 7.9.1 用户语义

运行已经结束，但结果尚未完全准备为用户可消费形态。

### 7.9.2 页面文案建议

- `Preparing results`
    
- `Packaging outputs`
    
- `Generating explanation`
    
- `Preparing downloadable artifacts`
    

### 7.9.3 推荐阶段提示

|阶段提示|用户理解|
|---|---|
|Packing results|正在整理结构化结果|
|Preparing artifacts|正在准备文件/工件|
|Generating explanation|正在准备解释文本|
|Finalizing report|正在准备结果展示对象|

### 7.9.4 页面表现

|页面|表现|
|---|---|
|Workbench Header|Current State = Processing Results|
|Analysis Tab|AI 可解释“为什么已经跑完但还不能下载”|
|Task Tab|显示 RESULT_PROCESSING / ARTIFACT_PROMOTING|
|Governance 页|作为主视图，展示当前后处理阶段|
|Report List|可出现 preparing 占位卡|
|Report Detail|若已创建入口，只展示 preparing 页面，不展示最终导出|

### 7.9.5 用户允许操作

- 查看任务详情
    
- 继续询问结果何时可见
    
- 等待结果完成
    

### 7.9.6 用户限制操作

- 不提前开放最终导出
    
- 不把“结果处理中”渲染成 Completed
    

---

## 7.10 Artifact Promoting 状态规格

如果系统将工件晋升建模为单独阶段，前端应将其视为 `Processing Results` 的一个子阶段，而不是新的用户主状态。

### 7.10.1 用户语义

系统正在准备或上传最终可交付产物。

### 7.10.2 页面文案建议

- `Preparing final outputs`
    
- `Final artifacts are being prepared`
    

### 7.10.3 页面表现

- Governance 页可显示为当前阶段提示
    
- Workbench Header 继续显示 `Processing Results`
    
- Results 页继续保持 preparing 状态
    

### 7.10.4 原则

- 不应让普通用户理解为一个全新业务状态
    
- 不应提前开放下载
    

---

## 7.11 Completed 状态规格

### 7.11.1 用户语义

结果已经进入可消费状态。

### 7.11.2 页面表现

|页面|表现|
|---|---|
|Workbench Header|Current State = Completed；主按钮 = View Results|
|Inputs Tab|显示本轮输入快照|
|Map|可加载结果图层|
|Analysis Tab|AI 进入结果解释模式|
|Governance 页|Lifecycle 完成；Artifacts 与 Result Bundle 可见|
|Report Detail|正式展示结果摘要、地图、指标、解释、导出|

### 7.11.3 用户允许操作

- 查看地图结果
    
- 查看结构化指标
    
- 导出结果
    
- 追问解释
    
- 返回场景继续新分析

## 7.12 运行期通知与提示规则

### 7.12.1 轻提示（Toast / Banner）适用场景

适合用轻提示的事件：

- Task submitted
    
- Entered queue
    
- Run started
    
- Run cancelled
    
- Results ready
    

### 7.12.2 不应只用轻提示的场景

以下状态不能只靠 toast，必须有页面主区域承载：

- WAITING_USER
    
- RUNNING
    
- PROCESSING_RESULTS
    
- FAILED
    
- CANCELLED
    

### 7.12.3 横幅提示建议

在 Workbench 顶部可使用状态横幅：

- warning：缺输入 / 可恢复失败
    
- info：queued / running / processing results
    
- success：results ready
    
- error：terminal failure
    

---

## 7.13 运行期 AI 说明规则

运行期间，AI 对话区的职责不是替代状态面板，而是补充解释。

### 7.13.1 Queued 时 AI 应回答

- 为什么还没开始
    
- 当前可以做什么
    
- 是否可以取消
    

### 7.13.2 Running 时 AI 应回答

- 当前运行到哪一步
    
- 为什么还没结果
    
- 当前是否还能修改输入（答案通常是否）
    

### 7.13.3 Processing Results 时 AI 应回答

- 为什么运行结束了还不能下载
    
- 还在准备什么
    
- 什么时候适合去看结果页
    

### 7.13.4 不应由 AI 独占承载的内容

- 权威状态
    
- 是否可 Resume
    
- 是否可 Cancel
    
- 是否已有正式结果可下载
    

这些仍必须由结构化 UI 显示。

---

## 7.14 推荐的最小实现顺序

如果你准备分阶段实现运行态 UI，推荐顺序如下：

1. 先落地 Queued / Running / Processing Results / Completed / Failed 五个主状态。
    
2. 给 Running 增加简单阶段提示：Preparing workspace / Running model / Packing results。
    
3. 增加 Cancelled / Cancelling。
    
4. 最后补 Heartbeat Lost / 状态不确定。

## 8. 使用建议

### 给设计师

- 优先看第 5 章和第 6 章，把状态流转画进页面原型。
    
- 不要只做静态页，要明确各状态下 Header、Task、Report 的变化。
    

### 给前端

- 先以主状态驱动页面，再加阶段说明。
    
- 不要让组件各自维护冲突状态，必须围绕统一任务状态联动。
    

### 给产品 / 架构

- 用本文约束“什么时候算完成、什么时候只是处理中”。
    
- 用本文作为 WAITING_USER、RUNNING、PROCESSING_RESULTS、FAILED 的评审基准。
